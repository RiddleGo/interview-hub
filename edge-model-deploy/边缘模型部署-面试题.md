# 边缘模型部署 · 面试题实录

面向 ONNX 导出、INT8、算子/NPU、上板验收这条链路的口头问答。下面按「先给结论 → 再展开你怎么干」写，不是背定义用的标准答案。

---

## 一、导出与图

### 1. 训练是 `.pt`，部署为什么要 ONNX，还要再转成「芯片模型」？

https://mp.weixin.qq.com/s/ZPKAA9syzIIewuZy2w5ULw

`.pt` 绑 PyTorch：算子实现、版本、自定义 op 都在那个 runtime 里，车机/相机 SoC 上往往装不了，或者装了也占内存、启动慢。

ONNX 是中间合同：图结构 + 权重 + 输入输出名，给 ORT 做 Golden，也给厂商 `atc` / `trtexec` / 自家 converter 吃。这一步不对，后面全是玄学。

芯片模型（`.om`、`.bin`、engine 等）是 **已经 lowering 过** 的东西：算子映射到 NPU 指令、layout 可能从 NCHW 改成 NHWC、该 fuse 的 fuse 了、该 tile 的 tile 了。PC 上 ORT 能跑，只说明图在数学上成立，不说明这颗 NPU 吃得下。

我面试会补一句：我们团队里 ORT FP32/FP16 是 **验收基准**，不是交付物；交付物是芯片侧二进制 + 版本锁定的工具链。

---

### 2. `[1, 3, 640, 640]` 各维什么意思？静态 / 动态 shape 怎么选？
https://mp.weixin.qq.com/s/Eo44jEg8T__G0Babwtk13g
`NCHW`：`1` batch，`3` RGB，`640×640` 高宽。导出时 `dynamic_axes` 不写就是静态；写了可能是 `batch` 或 `height/width` 动态。

**静态** 的好处很实在：编译器能做满内存规划，延迟抖动小，工具链支持最全。车载、安防固定机位，我默认静态，除非产品硬要多分辨率。

**动态** 的问题不在「能不能导出」，在 **芯片支不支持、支不支持得漂亮**。很多 NPU 对动态轴是：能编过，但每来一次新尺寸就重新走一遍 runtime 路径，或者干脆 fallback CPU。面试别只说「动态更灵活」，要说你会先翻 toolchain release note，再决定是「一个模型多尺寸」还是「3 个固定尺寸三份模型」。

非正方形（比如 `160×320`）在窄屏上像素数可以比 `640×640` 少一个数量级，这是业务侧优化，和「动态轴」不是一回事。

---

### 3. Opset 选错了会怎样？你怎么定？
https://mp.weixin.qq.com/s/DrJJIqlezZN0wtTgnkv8lg
Opset 是算子 **语义版本**。同一个 `Resize`，opset 11 和 13 属性不一样；`GridSample` 新老行为差过。

- 导出 opset **高于** 工具链：转换阶段报 `Unknown attribute` 或直接 unsupported。
- 导出 opset **过低**：PyTorch 新导出路径需要的算子根本没有。

我做法不背数字：看三处——`torch.onnx.export` 默认、ORT 该版本的 [算子表](https://github.com/microsoft/onnxruntime/blob/main/docs/Versioning.md)、芯片文档写的「支持 opset 范围」。导出后 `onnx.checker.check_model`，再 `grep` 一眼有没有 `aten::` 残留、有没有 `Constant` 炸图。YOLO 系常见 11–17 都见过，**以你当前 ORT + converter 组合实测为准**，别跟博客走。

---

### 4. PC 上 ONNX Runtime 和 NPU 上跑，差在哪？
https://mp.weixin.qq.com/s/buTfIAM_krkk1DzvLkZOdA
ORT 是通用 EP：CPU/CUDA/DML，图优化偏通用 fuse，精度 FP32/FP16 随意切。

NPU 是 **白名单 + 编译期决策**：哪些 op 能 int8、哪些必须 float、哪些要插 `Transpose`，都在 convert 日志里。同一张 ONNX，ORT 7ms，芯片 40ms，不稀奇——可能 int8 路径没走硬件 MAC，或者中间 feature 一直打 DDR。

所以验收必须 **同输入、同预处理、两端各跑一遍**。只比 mAP 不比 tensor，出了问题你不知道是量化还是 converter 把 `Resize` 搞成了近似实现。

---

### 5. 【场景】ORT 正常，转芯片报 `Unsupported operator: Softmax`

别先改模型，先定位 **哪一个 Softmax、前后接什么**。

1. Netron 点开节点，看输入 rank、是否在 `Sigmoid` 前、是不是 DETR 式 attention 里的那一颗。
2. 翻芯片算子表：有的只支持 `axis=-1` 且 `dim<128`，有的根本不支持，要拆 `Exp + ReduceSum + Div` 或查厂商给的 `Softmax` 替代层。
3. PC 上先换等价子图再 `check_onnx`，ORT 对一下输出；别等设备编译半小时才发现图错了。
4. 通过后记进算子库：`芯片X + Softmax(axis=1) → 用 xxx 替代`，下次同类头直接套。

这类题考的是：**unsupported 不等于模型废了**，是 graph surgery + 精度回归。真正费时间的是 attention 里 Softmax 和 scale 叠在一起，INT8 一换全崩，要单独留 FP16。

## 二、量化

### 6. 对称量化公式？scale 和 zero_point 各干什么？

```
q = saturate( round(x / scale) + zero_point )   # 对称时常 zp=0
x' = (q - zero_point) * scale
```

`scale` 是浮点里「一格整数」代表多少实数。`zero_point` 让某个浮点值 **正好** 落在整数格子上；激活经过 ReLU 后全正，非对称用 zp 把 0 对齐，避免大量「本该是 0」的数被 round 到 1 格以外。

对称实现简单、硬件友好；非对称在 **分布不关于 0 对称** 时更省误差。面试如果追问「为什么 ReLU 后还用非对称」，答：看校准直方图是否严重偏置，以及芯片是否支持 per-tensor asymmetric。

---

### 7. 校准集是什么？没标签怎么用？

就是一批 **和线上一样预处理** 的输入，几百张典型图打成 `npz`/`npy`，跑 forward 统计 activation 的 min/max 或直方图。不要标签，因为 PTQ 不训练权重，只定 scale。

坑我见得多的：

- 校准用 ImageNet 风格归一化，线上用 `0~255` 或 BGR——scale 全错，余弦还能凑合，mAP 直接废。
- 校准只有白天大图，夜视小人没进分布，暗光层饱和。
- 张数太少（十几张），outlier 一张图把 max 拉爆。

我会要求校准集 **和训练集同分布但不必同张图**，最好带季节/光照/距离的 checklist，版本号写进发布说明。

---

### 8. PTQ 和 QAT？项目里怎么选？

PTQ：模型训完了再量化，几天内能出 INT8，精度掉多少看任务，检测掉 1–2 个点不罕见。

QAT：训练图里插 fake quant，要数据、要算力、要调，一般能追回 FP32 大部分指标。

我默认 **PTQ 先上车**；掉点超业务线再 QAT 或混合精度，而不是一上来 QAT（周期和客户等不起）。

PTQ 里有个老问题：**激活 outlier**。一层里 99% 值在 `[-2,2]`，几个 token 到 50，per-tensor scale 被 50 撑大，其余全挤在 0 附近——余弦看着还行，分类/回归头已经糊了。处理顺序：换 Entropy/MSE 校准、per-channel（权重）、敏感层 FP16、再考虑 QAT。Transformer 比 CNN 更常栽在这。

---

### 9. per-tensor 和 per-channel？权重和激活怎么配？

per-tensor：整张量一个 scale，硬件最省事。

per-channel：通常沿 **输出通道** 每 channel 一个 scale，权重几乎总是 per-channel（ONNX 里 `axis=0`），避免「一个 channel 权重特别大把整张 scale 拉大」。

激活 per-channel 精度好，但很多 NPU **反量化代价高** 或编译器直接不支持，只能 per-tensor。面试要说清楚：**不是不想 per-channel activation，是芯片约束**；不行就 mixed precision 把敏感层 float 掉。

---

### 10. 余弦 99% 能上线吗？

余弦看的是向量夹角，对整体缩放不敏感。99% 在 **层输出对比** 里算「能往下查」的门槛，不是「能发货」的门槛。

我遇到过：全局余弦 99.6%，但检测 head 某个 stride 的 box 分支余弦 94%，画出来框全飘——因为平均被 backbone 的大 tensor 淹没了。

上线要三层：关键层余弦 / MSE、任务指标（mAP、误检率）、坏例图人工扫。余弦只回答「数值像不像」，不回答「业务对不对」。

---

### 11. QDQ 和 QOperator，和部署什么关系？

静态量化后两种图：

- **QDQ**：`QuantizeLinear` + 浮点算子 + `DequantizeLinear`，scale/zp 在图上看得见。TensorRT、很多 NPU converter 靠这个模式做 fusion，把 Q-D 吃掉变成真 int8 算子。
- **QOperator**：直接 `ConvInteger` 这类，ORT CPU 上有时反而慢，部分 EP 支持不完整。

做边缘交付，团队里口头说的「量化模型」多半是 **QDQ 形态的 ONNX**，再交给工具链。验收时对比 FP32 与 INT8，要确认 compare 的是 **Dequantize 之后** 还是 int8 tensor，别比错节点。

---

### 12. 【场景】INT8 后 mAP 45% → 38%，排查顺序

按这个顺序能少背锅：

1. **预处理对齐**（最高频）：letterbox 颜色、mean/std、`RGB/BGR`、`NCHW/NHWC`、resize 插值。用同一张图 dump 进 ORT 和进设备的 `input.bin`，字节级一致。
2. **校准集**：换一批、加大到几百张、覆盖坏例场景；看校准前后 activation histogram 是否合理。
3. **量化配置**：权重 per-channel；`ReduceMean`、`Softmax`、小通道 head 进 skip；试 `MinMax` vs `Entropy`。
4. **分层对比**：哪一层余弦先掉到 98% 以下，从那一层往前查是量化还是 fuse 错了。
5. **确认掉点发生在 ORT INT8 还是只在芯片**：若 ORT INT8 就好、芯片差，别继续调 calib，转去看 converter 日志和算子实现。

别一上来就 QAT，前两步能救回来一半项目。

## 三、算子与 NPU

### 13. 算子搜索在解决什么？

两件事缠在一起：

**兼容性**：图里有芯片没有的 op，或有限制（`Conv` kernel>7、`Resize` 只支持 nearest）。不解决，编译不过。

**性能**：同一个 `7×7 Conv` 可以换成 `3×3` 堆叠，数学近似，FLOPs 差不多，但在某颗 NPU 上利用率差一倍——这是 **搜实现**，不是搜超参。

工程上一般是：查算子库 → 自动 benchmark 几个等价子图 → 记结论。新人最容易跳过「查库」，重复踩同一颗芯片的 Softmax 坑。

---

### 14. Tiling 是什么？选坏了为什么更慢？

片上 SRAM 放不下整块 feature/weight 时，compiler 按通道或空间切块，算完一块再算下一块。

块太小：launch、同步、写回次数多，算力空转。块太大：放不进 cache，反复拖 DDR，带宽打满。Profiling 里会看到某个 `Conv` 耗时离谱但 MAC 利用率很低，多半 tiling 或 layout 有问题。

这类经验靠 **芯片 + 输入尺寸** 记，不是通用常数；知识库里写「H×W=640、C=256 时推荐 channel tile=16」比写空话有用。

---

### 15. Fusion 举例

`Conv + BN + ReLU` 合成一个 kernel：中间 feature 不写 DDR，少两次 launch。有的编译器还能把 `Add` 并进 residual。

算子搜索阶段有时会 **故意** 把图排成 compiler 爱 fuse 的顺序（BN 折叠进 Conv 权重等），和「换一个数学等价的 Conv 实现」是两层优化。

---

### 16. FLOPs 差不多，为什么 7×7 比 3×3 堆叠还慢？

FLOPs 只数乘加。真实时间 = 计算 / 利用率 + 访存。大核在部分 NPU 上矩阵单元吃不满（利用率 20% vs 40% 很常见），中间 buffer 大，DDR 占比高，还难和 BN fuse。

所以结构替换（大核拆小核、depthwise 改 pointwise+depthwise）常常是 **手工** 做的，别指望 converter 自动帮你变快。

---

### 17. 算子知识库一条记录写什么？

至少：`算子名`、`芯片型号/工具版本`、`支持精度`、`限制（axis、kernel、shape）`、`推荐替代`、`tiling 备注`、`踩坑案例（日期+模型）`。

价值在于 **可检索**，不是写 wiki。新人先搜「Softmax + 昇腾 310」有没有记录，再决定是否改图。

---

### 18. 【场景】INT8 芯片模型比 PC FP32 还慢

先排除对比不公平：PC 用 GPU FP32 batch=8，设备 INT8 batch=1，没有可比性。

再查：

- 芯片 INT8 是否真的走 MAC（老平台 int8 路径假量化，来回 cast）。
- 图里是否大量 `Transpose`、`Slice`、`Gather`，内存 bound。
- 是否有 op fallback CPU（日志里会有 `CPU fallback` 字样）。
- Tiling 极差导致 DDR 瓶颈。
- 动态 shape 导致每次 resize 重新编译或走慢路径。

用厂商 profiler 看 **算子耗时排行**，别猜。INT8 的意义是功耗和吞吐，不是「名字带 int8 就一定快」。

## 四、上板与验收

### 19. NPU 和 GPU 部署差在哪？

GPU：算子全、生态成熟，适合训练和大 batch 云推理。

NPU：固定图、低功耗、算子白名单，优化在 **编译期**（fusion、tiling、layout）。换一颗芯片，往往 ONNX 能复用，芯片模型和算子库要重做。

一句话给面试官：**GPU 卖通用，NPU 卖单位能耗下的确定性延迟**——前提是你接受它的约束。

---

### 20. 为什么双端对比必须同一份 `input.bin`？

输出差异只能来自：量化、converter、驱动实现。输入差一个字节，余弦没有意义，还会误判成「量化不行」去瞎调 calib。

做法是 PC 侧 ORT 跑完把输入和输出都落盘，`adb push` 同一份 input，pull 回 output 再比。预处理也在 PC 生成 input 时做完，设备侧 **不要再做一遍 resize**（除非你就是测端到端 pipeline，那两端都要同流程）。

---

### 21. ADB 上板流程（口述版）

`adb push` 模型和 `input.bin` → `adb shell` 跑厂商 infer 可执行文件（参数里写模型路径、输入输出路径）→ `adb pull output.bin` → PC 脚本算余弦/MSE。

注意权限、路径、ABI（arm64-v8a）、以及 infer 是否链了错误的 `.so` 版本。第一次联调卡在 `adb devices` 比卡在精度多。

---

### 22. 芯片转换工具一般在干什么？

算子映射到指令集；子图 fusion；按 SRAM 做 tiling；layout 转换（可能插 `Transpose`）；可选 INT8 权重重排。

输入要给清：ONNX 路径、输入 shape、目标 IP、精度模式。日志里的 warning（「某层 fallback」「某算子用近似」）要留档，后面精度扯皮用。

---

### 23. 【场景】余弦 99.8%，检测框全乱

我第一反应不是量化，是 **后处理和 head**：

- 余弦对的是整段输出 tensor 还是单分支？YOLO 多 stride，可能一个 stride 坏了平均仍高。
- `sigmoid`/`exp` 解码、anchor、NMS 阈值是否在 C++ 里和 Python 训练不一致（常见：letterbox 逆变换漏了）。
- 类别顺序、`conf` 阈值、输入尺寸和训练不一致。

然后才查：head 层是否该 skip quant、Softmax 是否被近似。教训：**业务指标门禁高于张量余弦**。

---

### 24. 混合精度 / skip_layers 什么时候用？

逐层对比发现某几层余弦 <98%，或 mAP 对某层敏感，就把这些层标成 FP16（工具链里 `skip_layers`、混合精度列表）。

代价：模型变大、带宽上去、延迟涨。要 profiling 看涨多少，和产品谈「掉 0.5 点 mAP vs 多 3ms」。

不是层数越多越好；曾见过 skip 了整个 backbone 最后几层，INT8 意义不大。

## 五、综合题

### 25. 从训练完到边缘上线，你走什么链路？

我习惯按阶段说，每段有 **可交付物和回滚点**：

导出 ONNX（静态、命名、checker）→ 算子/结构兼容（改图、查库）→ PTQ + 校准 → ORT 上先验 INT8 精度 → 转芯片模型 → 设备推理 → 同 input 双端对比 → 业务指标 + 坏例回归 → 锁版本（torch/onnx/opset/toolchain/driver）。

岗位不同负责段不同，但 **必须能判断问题落在哪一段**——否则量化工程师和驱动工程师会互相甩锅一周。

---

### 26. 客户要动态多分辨率，静态 640 不想用

第一步：问清 toolchain 是否支持 dynamic axis、性能曲线怎样。不支持就三选一：多份固定尺寸模型、统一 letterbox 到 640、或云端大模型+边缘小模型分工。

支持动态也要测 **最坏分辨率** 的延迟和精度，很多项目最后折成 2～3 档分辨率各一版，维护成本反而低。

---

### 27. 你怎么定「量化可上线」标准？

没有银弹，我会和客户对齐一张表：

- 张量：关键层余弦（例如 head >99%，backbone 最差 >98%），加 SNR/MSE 看极端层。
- 任务：mAP/召回/误检相对 FP32 掉点上限（安防和智驾差很多）。
- 系统：P99 延迟、峰值内存、功耗、温升。
- 坏例集：必须单独过一版，不能只看平均指标。

附带：校准集 hash、工具链版本、驱动版本写进 release note，半年后能复现。

---

### 28. FLOPs 减半，延迟减半吗？

不。内存 bound、低利用率、小算子 launch 开销都能让 FLOPs 白减。要以 **profiler 和固定 input 的 repeated benchmark** 为准，FLOPs 只用于粗筛结构。

---

### 29. 动态量化和静态 PTQ？

动态量化：推理时再算一部分 activation scale，ORT 里对 RNN/部分线性层常见，CNN 全图 INT8 加速未必赚。

静态 PTQ：校准集离线定好 scale，推理无额外统计，边缘 CNN **主流**。NPU 编译型方案基本都要静态图+静态 scale。

---

### 30. 用几分钟讲「边缘部署工程师」日常在干什么？

在拉齐四件事：**精度、延迟、功耗、能不能在这颗芯片上编过**。训练给的是大 FP32 图，客户要的是固定场景下 INT8 实时。中间每层都可能吃精度——导出算子变了、校准分布偏了、converter 近似了、后处理写错了。

日常大量时间在：**同一输入对齐 PC 和设备**、读 convert/profiler 日志、往算子库里记「这颗芯片这样改」。熟练之后，同类 YOLO 头 unsupported 从三天缩到半天，靠的是沉淀不是聪明。

## 附：几个容易答飘的点

| 别人爱问的 | 别答成 | 更稳的说法 |
|-----------|--------|-----------|
| 余弦 99% | 可以上线 | 只是层对齐门槛，还要看 mAP 和 head |
| INT8 | 一定更快 | 看 MAC 是否真 int8、是否 memory bound |
