# Tiling 选坏了，为什么 MAC 利用率只有 18%？

> **导读**：Profiling 里某层 Conv 耗时 9ms，MAC 利用率却不到 20%——多半是 Tiling 和访存出了问题，不是量化算错。  
> **阅读时间**：约 12 分钟  
> **适合谁**：模型能编过、上板跑得慢，正在看 profiler 里 MAC / DDR 指标的工程师

---

## Profiling 里那句「MAC 18%」

某次检测模型上昇腾 310，INT8 编过、精度也对，但 backbone 里一层 `3×3 Conv`（`1×256×640×640`）单算子就要 9ms。打开 profiler，MAC 利用率 18%，DDR 带宽 92%。

同事第一反应是「INT8 量化有问题」。我先把这条 Conv 的 FLOPs 算了一遍——和 FP16 差不多，计算量没凭空变大。再看 MAC：芯片标称 16 TOPS，这层实际每秒只跑约 2.6 TOPS，82% 的乘加单元在等数据。

**Tiling（算子切片 / 分块）** 就是把放不进片上 SRAM 的大计算，按 N/C/H/W 切成小块轮流算。块切错了，计算单元空转，MAC 利用率就会很难看。本文把 Tiling 是什么、怎么选块、怎么用 MAC 和带宽排查，一次讲清楚。

---

## Tiling 在部署链路里干什么

```
训练 .pt
  → 导出 ONNX
  → 图优化（BN 折叠、融合）
  → PTQ / 芯片编译          ← 编译器在这里决定 Tiling
  → .om / .engine 上板
  → Profiler 看 MAC / DDR   ← 本文诊断指标出在这里
```

Tiling 发生在 **编译器 lowering 阶段**：芯片算子库把一个大 Conv/MatMul 拆成多个 tile，每个 tile 的数据量要塞进几 MB 的片上 SRAM。它和算子融合绑在一起——融合后一颗 kernel 要同时放输入块、中间块、输出块，SRAM 预算更紧，Tiling 策略也会跟着变。

你改不了训练代码里的 shape，但可以通过 **padding 通道到硬件对齐粒度、换 layout、调编译选项** 影响 compiler 怎么切。

## 零基础先搞懂：MAC、FLOPs、片上 SRAM

**MAC（Multiply-Accumulate，乘加运算）** 是一次「乘 + 加」：`result += a × b`。Conv 和 MatMul 的内层循环本质上就是大量 MAC。AI 芯片标称的 **TOPS**（Tera Operations Per Second，每秒万亿次操作）通常指的就是 **每秒 MAC 次数**，不是 FLOPs。

**FLOPs（Floating Point Operations，浮点运算）** 把一次乘法或一次加法各算 1 次。因此 **1 MAC = 2 FLOPs**。一颗 NPU 标「INT8 16 TOPS」，等价于 FP16 约 32 TFLOPs——很多厂商两种标法混用，本质同一套硬件。

**片上 SRAM** 是芯片上几 MB 级的缓存，带宽远高于片外 **DDR**（内存）。Tiling 的全部目的，就是让每个 tile 的工作集 **尽量完整放进 SRAM**，算当前块时 DMA 预取下一块，计算和搬数并行。

上面那层 Conv 的计算量粗算：

```
输出元素数 × 核大小 × 输入通道
= 1×256×640×640 × 3×3 × 256
≈ 2359 亿次 MAC ≈ 4718 亿次 FLOPs
```

跑 9ms 的话，实际算力约 2.62 TOPS，相对 16 TOPS 标称值就是 ~16% MAC 利用率。

## Tiling 到底是什么

**Tiling（算子切片）**：当整张 feature map 或权重矩阵太大、塞不进 SRAM 时，编译器按 **通道（C）** 或 **空间（H/W）** 方向切成多块（tile），逐块计算再拼成完整输出。

直觉类比：案板只有这么大，一整块肉切不动，就 **分片切、分片炒**；切太碎——刀工和洗锅时间比炒菜还长；切太大——案板放不下，得反复从冰箱拿，同样慢。

Tiling 不改动数学结果（在数值精度允许范围内），只改 **访存路径和 kernel 调度粒度**。Profiling 里某个 Conv 耗时离谱、MAC 却很低，我第一反应总是查 **tile 是否过小（调度淹没计算）或过大（cache 颠簸、DDR 打满）**。

## MAC 利用率：比 FLOPs 更靠谱的「慢不慢」信号

**MAC 利用率 = 实际每秒 MAC 数 ÷ 芯片理论峰值 MAC 数 × 100%**

| 利用率区间 | 常见含义 | 下一步 |
|-----------|---------|--------|
| >70% | 计算单元忙，慢多半是真算力需求大 | 改结构、减 FLOPs、换更强芯片 |
| 30%～60% | 访存或 tile 偏大，数据供不上 | 看 DDR 带宽、缩小 tile、查 layout |
| <30% | 调度开销或 tile 过小 | 增大 channel tile、查对齐、减少 launch |

Conv/MatMul 占模型绝大部分算力，执行过程几乎全是 MAC。所以 **MAC 利用率高** 说明乘加单元在干活；**低** 说明在等 DDR、等同步、或 tile 切得不对——和 FLOPs 总数没有一一对应关系。

我见过 FLOPs 更低的「轻量模型」实机更慢：Tiling 差导致 MAC 20%、FLOPs 低的模型 MAC 70%，后者反而更快。

## 三种 Tiling：太小、太大、刚好

### Tile 太小 → 调度淹没计算

编译器默认把 256 通道按 **C=4** 一块切，要跑 64 个 tile。每个 tile 都要：启动 kernel → DDR 搬入 SRAM → 计算 → 写回 → 同步。单块计算只有几微秒，调度加搬数却要几十微秒。

**现象**：MAC 10%～20%，DDR 带宽也不高（<50%），profiler 里同一层出现大量细碎 sub-kernel。  
**原因**：launch 和同步次数过多，计算单元大部分时间在 idle。  
**怎么查**：看算子是否被拆成远多于「通道数 ÷ 对齐粒度」的块数。  
**怎么改**：增大 channel tile（如 16、32），对齐硬件粒度；能融合的先融合，减少中间落 DDR。

### Tile 太大 → cache 颠簸，DDR 打满

反过来把 256 通道 **整块一次算**，单张 feature 约 200MB，远超几 MB 的 SRAM。数据刚载入就被下一波覆盖，反复打 DDR。

**现象**：MAC 30%～40%，DDR 带宽 >90%。  
**原因**：工作集放不下 SRAM，memory-bound。  
**怎么查**：带宽曲线顶满、MAC 中等偏低。  
**怎么改**：让 compiler 切小空间维或通道维；检查是否与融合抢 SRAM 导致被迫大块。

### Tile 刚好 → 流水线满负荷

通道按 **32** 一块（对齐 16 的倍数），单 tile 工作集完整进 SRAM；算当前块时 DMA 预取下一块。

**现象**：MAC 70%～80%，同一层耗时从 9ms 降到 2～3ms 并不罕见（视芯片和 shape 而定）。  
**原因**：计算与访存并行，乘加单元持续有活干。

三种情况 **FLOPs 完全相同**，实机速度差 3～5 倍我见过，别背死倍数，以 profiler 为准。

## 昇腾 NPU：对齐 16 不是玄学

昇腾达芬奇架构的 **Cube** 单元是 16×16×16 的 MAC 阵列，每周期 4096 次 MAC。昇腾 310 有 4 个 Cube，1GHz 下理论 INT8 约 16 TOPS——和标称一致。

**通道数、tile 大小最好对齐 16 的倍数**。C=255 时，最后一个 16×16 块填不满，部分 MAC 单元空转，利用率直接掉一截。工程上常见做法：导出前把通道 **pad 到 16 的倍数**，或让 compiler 做 channel padding。

**Layout** 也影响 Tiling 效果。昇腾原生 **NC1HWC0**；若图里是 NCHW，编译器可能先插 Transpose，MAC 利用率掉一半以上并不奇怪。导出 ONNX 时尽量用芯片友好的 layout，或在图优化阶段改掉多余 Transpose。

Cube 跑 Conv/MatMul 的同时，Vector 单元可并行做 BN、Bias、ReLU——Tiling 合理时，融合链更容易吃满流水线。

## 最常见坑（附排查顺序）

### 坑 1：FLOPs 低就一定快

**典型做法**：选 FLOPs 更小的 backbone，未看上板 profiler。  
**现象**：比旧模型 FLOPs 少 30%，延迟反而更长。  
**原因**：新结构 tile 不友好（奇数通道、大 kernel、难融合），MAC 利用率差。  
**怎么查**：同输入尺寸对比两层 Conv 的 MAC%、耗时。  
**怎么改**：以 **MAC 利用率 + 端到端延迟** 选型，不单看 FLOPs。

### 坑 2：量化后一定更快

**典型做法**：INT8 编过即认为提速。  
**现象**：INT8 路径 MAC 从 80% 掉到 20%，比 FP16 还慢。  
**原因**：INT8 算子 Tiling 路径与 FP16 不同，或多了 Cast/Transpose。  
**怎么查**：分别 profile FP16 与 INT8 同层的 MAC、DDR。  
**怎么改**：查算子是否真走 INT8 MAC；改 layout、调 tile；必要时混合精度。

### 坑 3：MAC 100% 就等于优化到头

**典型做法**：看到 MAC 100% 就停手。  
**现象**：MAC 100% 且 DDR 也 100%，延迟仍不达标。  
**原因**：memory-bound，计算满负荷但数据路径仍是瓶颈。  
**怎么查**：同时看 MAC 与 DDR 曲线。  
**怎么改**：融合减 DDR 往返、优化 Tiling、减 feature 尺寸——不是再加算力。

### 坑 4：忽略融合与 Tiling 抢 SRAM

**典型做法**：只调 Tiling，图里 Conv-BN-ReLU 仍三颗独立 kernel。  
**现象**：tile 怎么调 MAC 都上不去。  
**原因**：中间 feature 反复写 DDR，融合没成立，SRAM 预算被浪费在搬数上。  
**怎么查**：dump 编译后图，看是否 fused op。  
**怎么改**：BN 折叠、去掉链中 Transpose/Cast，再调 Tiling。

## 怎么用 MAC 和带宽快速排查

某层 Conv 耗时异常时，我按这个顺序看：

1. **MAC 利用率**  
   - <30%：优先怀疑 tile 过小或 layout 转置  
   - 30%～60%：tile 偏大或 memory-bound  
   - >70%：计算 bound，考虑改结构或接受算力上限  

2. **DDR 带宽利用率**  
   - >90%：减访存——融合、更小 tile、更好 layout  
   - <50% 且 MAC 也低：调度 bound，增大 tile、减 launch 次数  

3. **数据布局与对齐**  
   - NCHW vs NC1HWC0、通道是否 16 对齐、是否有 CPU fallback  

4. **对比同 shape 不同精度**  
   - INT8/FP16 同层 MAC 差异大，常是路径或 Tiling 策略不同  

经验值 **因芯片、输入尺寸、compiler 版本而异**。知识库里记「H×W=640、C=256、昇腾 310、channel tile=32 时 MAC ~75%」比背 universal 常数有用。

## 面试追问

### 1

**问：** Tiling 是什么？选坏了为什么更慢？

**答：** 片上 SRAM 放不下整块 feature 或权重时，编译器把 Conv/MatMul 按通道或空间切成小块轮流算。块太小，launch 和同步次数多，MAC 空转；块太大，放不进 cache，反复拖 DDR，带宽打满。Profiling 里某层耗时高、MAC 利用率低，多半 tiling 或 layout 有问题，不是算子数学错了。

### 2

**问：** MAC 利用率和 FLOPs 有什么关系？

**答：** FLOPs 是静态计算量，MAC 利用率是运行时乘加单元忙闲比例，二者没有一一对应。同样 FLOPs，Tiling 好可能 MAC 70%、Tiling 差可能 20%，延迟差几倍。排查性能应先看 profiler 里的 MAC 和 DDR，再决定是改 Tiling、融合还是改结构。

### 3

**问：** 为什么 INT8 模型 MAC 利用率只有 18%，不一定是量化精度问题？

**答：** 18% 说明芯片大部分乘加单元在等数据，是 memory-bound 或调度 bound。量化精度问题通常体现在 mAP 掉点，不会单独把 MAC 打到 18%。先查该层 tile 是否过碎、DDR 是否打满、layout 是否多了一次 Transpose，再查 calibration。

### 4

**问：** 昇腾上 Tiling 为什么要对齐 16？

**答：** Cube 单元是 16×16×16 的 MAC 阵列，通道和 tile 不对齐 16，最后一个块填不满，部分 MAC 闲着。工程上把通道 pad 到 16 倍数，或让 compiler 做 padding，避免尾部空转。

### 5

**问：** Tiling 和算子融合是什么关系？

**答：** 融合把 Conv-BN-ReLU 焊成一颗 kernel，中间结果不落 DDR；但 fused kernel 要在 SRAM 里同时放输入块、中间块、输出块，SRAM 预算更紧，Tiling 必须一起算。融上了，有效 tile 可以更大、DDR 次数更少；融不上，每段中间结果都要单独 Tiling 和搬数，MAC 很难高。

## 附录 A：Tiling / 性能排查 checklist

```
[ ] 已用 profiler 定位 TOP 耗时算子，而非只看端到端延迟
[ ] 已记录该层 MAC 利用率、DDR 带宽利用率、单算子耗时
[ ] MAC <30% 时已检查 tile 是否过碎（sub-kernel 数量异常多）
[ ] MAC 30%～60% 且 DDR >90% 时已检查 tile 是否过大、cache 颠簸
[ ] 通道数已对齐芯片粒度（昇腾常见 16 倍数）
[ ] 已确认 layout（NCHW / NC1HWC0），无多余 Transpose
[ ] 已 dump 编译后图，Conv+BN+ReLU 等融合链是否成立
[ ] INT8 与 FP16 同层已对比，排除假 INT8 或 Cast 路径
[ ] 日志无 CPU fallback 或未支持算子
[ ] 动态 shape 未导致每次 resize 重编译走慢路径
[ ] 已记录「芯片型号 + 输入 shape + 有效 tile 参数」供团队检索
[ ] 优化后以业务指标（mAP / 延迟 SLA）验收，不单看 MAC 数字
```

## 附录 B：粗算 Conv MAC 与利用率

下面脚本不依赖厂商 SDK，用来理解「9ms + 18% MAC」这类数字怎么来的；复制即可运行。

```python
def conv_macs(n, c_in, h, w, c_out, kernel=3):
    """单 batch Conv2d MAC 次数（不含 bias）。"""
    out_elems = n * c_out * h * w
    return out_elems * (kernel * kernel * c_in)


def mac_utilization(macs, latency_s, peak_tops):
    """
    macs: 该层总 MAC 次数
    latency_s: 该层耗时（秒）
    peak_tops: 芯片标称峰值 TOPS（按 MAC 计）
    """
    actual_tops = macs / latency_s / 1e12
    pct = actual_tops / peak_tops * 100
    return actual_tops, pct


if __name__ == "__main__":
    # 与正文示例一致：1x256x640x640, 3x3, 256->256
    macs = conv_macs(1, 256, 640, 640, 256, kernel=3)
    latency_ms = 9.0
    peak_tops = 16.0  # INT8 标称

    actual, util = mac_utilization(macs, latency_ms / 1000, peak_tops)
    print(f"MAC 次数: {macs/1e9:.2f} G")
    print(f"耗时 {latency_ms} ms → 实际 {actual:.2f} TOPS")
    print(f"MAC 利用率（相对 {peak_tops} TOPS）: {util:.1f}%")
    print()
    # 优化后假设
    latency_ms2 = 2.4
    actual2, util2 = mac_utilization(macs, latency_ms2 / 1000, peak_tops)
    print(f"Tiling 优化后 {latency_ms2} ms → {util2:.1f}% MAC, {actual2:.2f} TOPS")
```

## 术语速查

| 术语 | 全称 / 含义 | 一句话直觉 |
|------|------------|-----------|
| Tiling | 算子切片 / 分块 | 大算子切成小块塞进 SRAM，逐块算完再拼 |
| MAC | Multiply-Accumulate | 一次乘加；TOPS 通常按 MAC 计 |
| FLOPs | Floating Point Operations | 静态运算次数；1 MAC = 2 FLOPs |
| MAC 利用率 | — | 乘加单元实际忙闲比；Tiling 差时会很低 |
| SRAM | 片上静态存储 | 几 MB 快缓存；Tiling 块要能放进去 |
| DDR | 片外内存 | 带宽有限；反复读写这里就 memory-bound |
| Cube | 昇腾矩阵单元 | 16×16×16 MAC 阵列；Tiling 常对齐 16 |
| NC1HWC0 | 昇腾友好 layout | 不对齐 NCHW 可能多一次 Transpose |
| Memory-bound | 内存瓶颈 | 算力闲着等数据，MAC 低、DDR 高 |
| Fusion | 算子融合 | 与 Tiling 共享 SRAM 预算，一起决定性能 |
