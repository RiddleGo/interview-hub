# 量化可上线标准，怎么和客户对齐？

> **导读**：余弦 99% 不能当发货门禁；真正可上线的 INT8，是张量、业务、系统、坏例四个维度一起过线，而且阈值要在量化开始前就和客户白纸黑字对齐。  
> **阅读时间**：约 28 分钟  
> **适合谁**：做过 PTQ、马上要交付 INT8 版本，或面试被问到「你怎么定量化上线标准」的工程师。

---

## 决策卡点：调了一个月，客户说不算

某次安防检测项目，团队按内部习惯把 INT8 推上线：head 余弦 99.2%，backbone 最差层 98.5%，测试集 mAP 从 0.86 掉到 0.855，整体只掉了 0.5 个点。内部评审觉得「够用了」。

客户验收时拿出自己的坏例集——逆光人脸、远处小目标、夜间红外——召回率掉了 8%，还有几类以前没见过的误检。客户原话：「你们平均指标好看，我关心的场景全崩了，这版不能发。」

复盘才发现三件事没做：量化前没和客户对齐容忍度；验收只看整体 mAP，没拆小目标和难场景；坏例集没单独跑。张量余弦和业务指标各看各的，系统侧也没证明 INT8 比 FP16 快了多少——白干一个月。

这类扯皮我见得不少：**量化能不能上线，从来没有「余弦 ≥ 99% 就合格」的银弹**。标准必须可量化、可复现、可回溯，而且越早对齐越好。

---

## 上线门禁在部署链路里干什么

```
  .pt → ONNX → ORT 验收（数学正确）
              │
              ▼  PTQ + 校准集
  INT8 ONNX（可选中间态）
              │
              ▼  converter（ATC / trtexec 等）
  芯片模型 (.om / .engine)
              │
              ▼  双端张量对比（余弦 / MSE / SNR）
  ┌──────────────────────────────────────────────────────┐
  │ 四维上线门禁（本篇）                                    │
  │ ① 张量级：层对齐、分分支，排除低级量化错误              │
  │ ② 业务级：mAP / 召回 / 误检，一票否决                  │
  │ ③ 系统级：延迟、内存、功耗，证明量化有价值              │
  │ ④ 坏例集：极端场景不断崖，防「平均好看、局部崩盘」      │
  └──────────────────────────────────────────────────────┘
              │
              ▼  Release Note + 归档（工具链 / 校准集 / 驱动）
  可复现、可回滚的正式版本
```

整条链里，本篇卡在 **「张量对比过了」和「客户签字收货」之间**。张量对比回答「算得对不对」；四维门禁回答「这版值不值得发、发了会不会线上翻车」。两者不能互相替代。

## 零基础先搞懂：本篇会反复出现的名词

### PTQ（Post-Training Quantization，后训练量化）

训练完的浮点模型，用校准集统计各层数值范围，把 FP32/FP16 映射到 INT8。部署里多数项目走 PTQ，因为周期短。量化后的精度损失，要在本篇四个维度里分别验收，不能只看一个余弦。

### 余弦相似度（Cosine Similarity）

两个同 shape 张量拉平为向量 `a`、`b`：

    cos(a, b) = dot(a, b) / (norm(a) × norm(b))

`dot` 是点积，`norm` 是 L2 范数；实现时分母可加 `1e-12` 防除零。余弦看「方向像不像」，对整体缩放不敏感——**它是筛查工具，不是发货门禁**。检测 head 全局余弦 99.8%，某个 stride 分支只有 0.93 的情况很常见。

### MSE（Mean Squared Error，均方误差）

    mse = mean((a - b) ** 2)

逐元素差的平方再平均。余弦高但 MSE 大，往往是少数位置差很多（高 conf 区域、Softmax 尖峰处）。张量验收建议余弦 + MSE 一起看。

### SNR（Signal-to-Noise Ratio，信噪比）

量化噪声相对有效信号强度的度量，常用 dB 表示。余弦对「大部分点对、少数点极差」不敏感；SNR 能补这块。工程上我会看关键层 SNR 是否明显偏低——通常输出层希望 ≥ 20dB，整体最差层 ≥ 15dB，具体视任务调整。

### mAP / 召回率 / 误检率

**mAP（mean Average Precision）**：检测任务综合精度，测试集上算。**召回率**：该检出的检出了多少——漏检敏感场景（工业缺陷、智驾）往往卡这条。**误检率**：不该报的报了多少——安防误报多会惹投诉。业务指标是 **一票否决项**：张量再好看，业务不达标也不能发。

### P99 延迟

把单次推理耗时从小到大排序，第 99 百分位的值。平均延迟好看但 P99 尖峰高，线上 SLA 照样违约。量化验收要在 **最大分辨率、满负载、连续跑一段时间** 后测，不能测空载理想值。

### 校准集（Calibration Set）

PTQ 用来统计 activation 范围的样本集，**不参与训练、也不等于测试集**。校准集分布偏了，张量余弦和业务指标都会歪。Release Note 里必须记校准集来源、张数、MD5，否则半年后没法复现。

### 坏例集（Bad Case Set）

从历史线上问题、边界场景、长尾样本里单独整理的集合，**不参与平均指标统计，但必须单独过一遍**。平均 mAP 掉 0.5% 但某类逆光场景掉 30%，就是坏例集要拦住的。

## 四维门禁到底是什么

没有银弹，我会和客户对齐一张表，四个维度同时约束：

| 维度 | 回答什么问题 | 在门禁里的权重 |
|------|-------------|---------------|
| 张量级 | 量化有没有引入结构性数值错误 | 底线：不过则业务大概率不过 |
| 业务级 | 用户能感知的效果够不够 | 核心：一票否决 |
| 系统级 | 量化带来的性能收益够不够 | 价值线：否则不如 FP16 |
| 坏例集 | 极端场景会不会翻车 | 保险：防局部崩盘 |

核心原则就一句：**精度损失在业务容忍范围内，性能收益达到预期，极端场景不出现断崖式劣化。** 所有阈值最终以业务线为准，下文的数字是工业界常见参考，签合同时要改成你们和客户约定的具体值。

---

## 第一维：张量级精度——底层一致性兜底

张量精度是基础门槛，不是最终验收标准。它的作用是 **快速排除低级错误**：校准集覆盖不对、某层 scale 算飞、converter 插了意外 Cast、比错输出节点——这些张量层一眼能看出来。张量过了不代表业务一定过；张量不过，业务几乎肯定会出问题。

### 分层设阈值，不搞一刀切

不同层对业务的影响差很多。检测模型里 head 直接影响框和 conf，backbone 误差有时会被后续层部分抵消。参考多数项目的做法：

| 层级类型 | 余弦相似度（参考） | 说明 |
|---------|------------------|------|
| 输出层 / 检测头 / Head | ≥ 99% | 误差直接传导到框和类别 |
| Backbone 卷积层 | 最差层 ≥ 98% | 看最差层，不看平均 |
| 激活 / 归一化等轻量算子 | ≥ 97.5% | 数值范围特殊，可略放宽 |

表后补一句硬约束：**任何一层余弦低于 95% 都应先停下来排查**，多半是校准集不够或算子实现有问题，别带着明显失真层进业务验收。

### 多指标互补

只用余弦会掩盖极端值误差。我会搭配：

- **SNR**：看有效信号和量化噪声的比值，抓「大部分准、少数极差」
- **MSE**：看绝对误差量级，和余弦配合判断

### 校验规则：拆细了比

两个坑我见得尤其多：

**全局平均掩盖分支问题。** YOLO 类模型 P3/P4/P5 三个 stride 分支要分别算余弦和 MSE，不能整段 head 展平算一个数。P3 元素量往往是 P5 的十几倍，全局 99.x% 可能把 P3 的 0.93 洗成「过关」。

**背景把指标洗好看。** 检测特征图里背景格占绝大多数，低值区域余弦很高，但高 conf 前景格已经偏了。可选做法：只在 conf 高于阈值的格子上单独算 MSE，和全局 MSE 对比（附录 B 有脚本）。

## 第二维：任务级业务指标——一票否决

这是最高优先级。张量再好看，mAP / 召回 / 误检不达标也不能上线；张量略低，业务完全满足要求，在客户认可的前提下可以接受。

### 没有通用阈值，按场景对齐

安防和智驾的容忍度差一个数量级不稀奇。量化 **开始前** 就要和产品 / 客户对齐，写进验收文档：

| 业务场景 | 核心指标 | 可接受掉点（参考） | 说明 |
|---------|---------|------------------|------|
| 自动驾驶 / 智驾感知 | 目标召回率 | ≤ 0.3% | 安全相关，宁慢勿错 |
| 工业缺陷检测 | 缺陷召回率 | ≤ 0.5% | 漏检成本远高于误检 |
| 安防监控分析 | mAP / 准确率 | ≤ 1.0% | 整体精度要求高 |
| 通用零售 / 客流统计 | 准确率 | ≤ 2.0% | 容错空间大，偏性能和成本 |
| 消费级娱乐应用 | 主观效果 | ≤ 3%～5% | 人眼无明显差异即可 |

表后：**最终以业务线为准。** 客户如果只关心「小目标召回」，你就不能拿整体 mAP 0.8% 的掉点去交差。

### 不能只看平均，必须细分维度

整体 mAP 只掉 0.8% 看起来达标，小目标召回掉 5% 照样翻车——我踩过。验收至少要拆：

- **按目标大小**：大 / 中 / 小目标分别统计
- **按难易场景**：正常光、暗光、逆光、遮挡、密集
- **按错误类型**：漏检增加还是误检增加，业务代价完全不同

## 第三维：系统级性能——量化的价值底线

做量化如果只盯精度，忘了初衷是 **更快、更省内存、更省电**，那 INT8 往往不值得维护。系统维度和精度维度同等重要。

### 延迟：平均 + P99，测最坏场景

- **平均延迟**：相对 FP16，提升 ≥ 30% 在多数项目里才算「量化值得」（视芯片和模型而定；提升不足 20% 又没功耗压力，我会建议直接用 FP16）
- **P99 延迟**：控制在平均延迟的 120% 以内，避免偶发尖峰
- **测试前提**：最大输入分辨率、满负载、连续运行 10 分钟后再测

### 内存与体积

- INT8 权重体积通常是 FP16 的一半左右；偏差很大说明大量层 skip 了量化
- 运行时峰值 DDR 占用应低于 FP16，至少不能更高——边缘设备内存紧张时这是硬线

### 功耗与温升（边缘设备必测）

电池供电或散热紧的产品，我会把功耗放在延迟同级甚至更高：连续推理稳态功耗相对 FP16 应有明显下降；连续跑 30 分钟，芯片温升不超过产品规格（常见 ≤ 40℃ 量级，以硬件手册为准）。

---

## 第四维：坏例集专项——守住线上稳定性

平均指标达标，不代表极端场景没问题；线上投诉往往来自那百分之几的坏例。坏例集不参与平均统计，但必须 **单独跑一遍**。

### 坏例集怎么构成

- **历史线上 Bad Case**：以前出过问题的典型场景，确保不回归
- **边界极端场景**：最小 / 最大目标、极端光照、极端角度、密集遮挡
- **长尾分布**：出现频率低，一旦出错影响大的样本
- **临界样本**：容易误检、容易漏检的「擦边球」

### 验收标准（参考）

1. **零新增恶性错误**：不能出现 FP16 没有的、严重影响业务的新错误类型
2. **坏例集指标不断崖**：坏例集上的掉点幅度，不应显著高于平均测试集（比如平均掉 0.5%，某类坏例掉 30% 就不能发）
3. **允许同步小幅下降**：坏例整体随模型同步略降可以接受，但不能某类场景完全失效

---

## 交付底线：版本锁定，半年后能复现

量化结果和工具链、校准集、驱动版本强相关。不锁版本，半年后出问题根本说不清是谁的锅。正式 Release Note 和归档里，我至少会写：

| 类别 | 必记字段 |
|------|---------|
| 模型 | 原始版本、权重 commit、ONNX Opset |
| 量化 | 工具名 + 精确版本、算法（KL / MinMax 等）、校准集 MD5 + 张数 + 来源、`skip_layers` 列表 |
| 编译 | converter 版本、编译参数、融合配置、AutoTune 结果 |
| 环境 | 驱动 / 固件、运行时库版本 |
| 验收 | 四维测试数据、阈值达标情况、坏例记录 |

归档文件建议成套留：**FP16 ONNX、INT8 ONNX、芯片离线模型、校准集清单、量化 / 编译配置、验收报告**。目标很简单——半年后能 100% 复现当前环境与结果。

## 最常见坑（附排查顺序）

### 坑 1：量化做完才和客户谈标准

典型做法：内部按「余弦 99%」调一个月，交付时客户用另一套测试集和阈值。  
现象：平均指标「达标」，客户不签字。  
原因：容忍度、核心指标、坏例定义从未对齐。  
怎么查：翻邮件和会议纪要，看有没有书面阈值。  
怎么改：**量化开始前** 开对齐会，四维阈值写进验收文档；张量 / 业务 / 系统 / 坏例各写一行「达标 / 不达标」。

### 坑 2：全局余弦过关，某分支已崩

典型做法：整段 head 展平算一个余弦。  
现象：全局 99.x%，mAP 却掉很多。  
原因：大分支把坏分支平均掉了。  
怎么查：按 P3/P4/P5 或各输出节点拆开算（附录 B）。  
怎么改：分分支验收；任一分支明显偏低先查该分支量化或 skip 策略。

### 坑 3：只看 mAP，不拆小目标和难场景

典型做法：测试集整体 mAP 掉 0.5% 就盖章。  
现象：客户关心的逆光 / 小目标场景召回崩了。  
原因：平均指标掩盖局部劣化。  
怎么查：按尺寸、光照、遮挡维度重算指标。  
怎么改：验收文档里写明细分维度阈值；坏例集单独统计。

### 坑 4：精度过了，性能没提升

典型做法：INT8 只调校准和 skip，不测延迟和内存。  
现象：上线后老板问「怎么比 FP16 还慢」。  
原因：大量 Cast / CPU fallback，或内存型算子占比高，INT8 没吃上 MAC。  
怎么查：profiler 看算子耗时和精度列；对比 FP16 同输入同 batch。  
怎么改：性能不达标就查编译日志和混合精度配置；收益不足 20% 且无功耗刚需时，考虑维持 FP16。

### 坑 5：没归档校准集和工具链版本

典型做法：模型发了，配置散落在个人电脑。  
现象：半年后客户投诉，换了一台机器复现不出当时结果。  
原因：校准集、converter、驱动任一变了，输出都会变。  
怎么查：对照 Release Note 字段是否齐全。  
怎么改：交付时成套归档；校准集算 MD5 写进 release note。

## 和客户对齐：建议流程

1. **立项 / 量化启动前**：拿四维表和客户过一遍，改参考阈值为合同数值，确认测试集和坏例集归属谁提供。
2. **量化迭代中**：张量维先过关再投入大量业务测；每版记录工具链版本，别在旧版上比新版业务指标。
3. **预发布**：四维 checklist 逐项打勾（附录 A）；坏例集单独出报告，不混进平均 mAP 幻灯片。
4. **正式发布**：Release Note + 归档；客户签字页引用同一份阈值表，避免口头理解不一致。

面试口述时我会浓缩成一句：**张量看层和分支，业务一票否决，系统证明值得量化，坏例防崩盘，版本全锁进 release note。**

## 面试追问

### 1

**问：** 你怎么定「量化可上线」标准？

**答：** 没有银弹，我会和客户对齐一张四维表：张量维关键层余弦（head 通常 ≥ 99%，backbone 最差 ≥ 98%）加 SNR/MSE 看极端层；任务维 mAP/召回/误检相对 FP32 的掉点上限，安防和智驾差很多；系统维 P99 延迟、峰值内存、功耗温升；坏例集必须单独过一版，不能只看平均。校准集 hash、工具链和驱动版本写进 release note，保证半年后能复现。

---

### 2

**问：** 张量余弦 99% 了，能不能上线？

**答：** 不能单凭余弦发货。余弦是排查工具，不保证 mAP——全局平均还可能掩盖坏分支。我会再看业务指标、细分场景、坏例集和系统收益。张量过了业务不过，不能上；业务过了，关键层余弦一般也过得去。

---

### 3

**问：** 业务指标和张量指标哪个说了算？

**答：** 业务指标一票否决。mAP、召回、误检、坏例图是客户真正关心的；张量余弦/MSE 用来快速定位是量化问题、converter 问题还是后处理问题。面试和项目里都要把话说清：张量是底线，业务是门禁。

---

### 4

**问：** 坏例集和测试集有什么区别？

**答：** 测试集算平均指标，看整体掉点是否在容忍范围内；坏例集专门收集历史线上问题、边界和长尾场景，不参与平均，但要单独验收，防止「平均好看、局部崩盘」。很多客户投诉就来自坏例集没单独跑的那几类场景。

---

### 5

**问：** 量化后性能提升不明显，还发 INT8 吗？

**答：** 看产品约束。若相对 FP16 延迟提升不到 20% 左右、又没有内存或功耗硬需求，维护混合精度的成本往往不值得，我会建议继续 FP16。若有功耗或体积硬指标，即使加速有限也可能必须 INT8——这就得在立项时和客户把系统维阈值讲清楚。

[ ] 张量：backbone 最差层余弦 ≥ 约定值（参考 98%）
[ ] 张量：全局最低层余弦 ≥ 95%（硬红线，低于先排查）
[ ] 张量：关键层 SNR 达标（参考：输出层 ≥ 20dB，最差层 ≥ 15dB）
[ ] 张量：检测模型按 stride 分支分别算余弦 + MSE，不用全局平均
[ ] 张量：可选高 conf 区域单独 MSE，对比全局 MSE

## 附录 B：分层张量验收脚本（余弦 + MSE + SNR）

用于第一维验收：按层或按分支对比 Golden 与 INT8 输出。`SHAPES` 须与 Netron 里输出节点一致；若只有整段输出，只填一项即可。

```python
import numpy as np

# ---------- 按项目修改：每项 shape 对应一个输出或分支 ----------
SHAPES = [
    (1, 3, 80, 80, 85),   # 示例：P3
    (1, 3, 40, 40, 85),   # P4
    (1, 3, 20, 20, 85),   # P5
]
NAMES = ["P3/stride8", "P4/stride16", "P5/stride32"]
GT_PATH = "output_gt.bin"      # PC Golden（FP16/FP32）
INT8_PATH = "output_int8.bin"  # INT8 侧同 input 推理结果

# 分层阈值（与合同约定一致，此处为参考）
THRESH = {
    "head_cos": 0.99,
    "backbone_cos_min": 0.98,
    "global_cos_floor": 0.95,
    "snr_db_min": 15.0,
}


def load_parts(path, shapes):
    flat = np.fromfile(path, dtype=np.float32)
    parts, off = [], 0
    for sh in shapes:
        n = int(np.prod(sh))
        parts.append(flat[off : off + n].reshape(sh))
        off += n
    if off != flat.size:
        raise ValueError(f"{path}: 元素数 {flat.size} 与 shapes 合计 {off} 不符")
    return parts


def cosine_mse(a, b):
    af = a.flatten().astype(np.float64)
    bf = b.flatten().astype(np.float64)
    cos = np.dot(af, bf) / (np.linalg.norm(af) * np.linalg.norm(bf) + 1e-12)
    mse = np.mean((a.astype(np.float64) - b.astype(np.float64)) ** 2)
    return float(cos), float(mse)


def snr_db(signal, noise):
    """signal、noise 为同 shape 数组；noise = signal - reference"""
    sp = np.mean(signal.astype(np.float64) ** 2)
    npow = np.mean(noise.astype(np.float64) ** 2)
    if npow < 1e-20:
        return float("inf")
    return 10.0 * np.log10(sp / npow)


def check(name, cos, mse, snr):
    ok = []
    if "P3" in name or "P4" in name or "P5" in name or "head" in name.lower():
        ok.append(cos >= THRESH["head_cos"])
    ok.append(cos >= THRESH["global_cos_floor"])
    ok.append(snr >= THRESH["snr_db_min"] or snr == float("inf"))
    status = "PASS" if all(ok) else "FAIL"
    print(f"  {name}: cos={cos:.6f}  mse={mse:.6e}  snr={snr:.2f}dB  [{status}]")
    return all(ok)


if __name__ == "__main__":
    gt_parts = load_parts(GT_PATH, SHAPES)
    int8_parts = load_parts(INT8_PATH, SHAPES)
    names = NAMES if len(NAMES) == len(SHAPES) else [f"out_{i}" for i in range(len(SHAPES))]

    print("=== 分分支 / 分层张量验收 ===")
    all_ok = True
    cos_list = []
    for name, g, n in zip(names, gt_parts, int8_parts):
        cos, mse = cosine_mse(g, n)
        snr = snr_db(g, g - n)
        cos_list.append(cos)
        all_ok &= check(name, cos, mse, snr)

    worst_cos = min(cos_list)
    print(f"\n最差层余弦: {worst_cos:.6f}  （参考 backbone 最差 ≥ {THRESH['backbone_cos_min']}）")
    if worst_cos < THRESH["backbone_cos_min"]:
        print("  提示：最差层低于 backbone 参考线，建议排查校准集或该层量化配置")
    print(f"\n汇总: {'四维之张量维 — 通过' if all_ok else '四维之张量维 — 存在未达标项，继续排查'}")
```

用法：双端用同一份 `input.bin` 分别跑出 `output_gt.bin` 和 `output_int8.bin`，改好 `SHAPES` 后执行 `python tensor_gate.py`。张量维过关后，再跑业务测试集、profiler 和坏例集。

## 术语速查

| 术语 | 含义 | 直觉 |
|------|------|------|
| 四维门禁 | 张量 + 业务 + 系统 + 坏例 | 单点余弦不能代表可上线 |
| 余弦 | Cosine Similarity | 筛查数值对齐，非发货标准 |
| MSE | 均方误差 | 抠元素级偏差 |
| SNR | 信噪比 | 补余弦对极端值不敏感 |
| mAP | 检测平均精度 | 业务核心指标之一 |
| 召回率 | 检出比例 | 漏检敏感场景硬线 |
| P99 | 第 99 百分位延迟 | 防平均延迟骗人 |
| 校准集 | PTQ 统计用样本 | 须记 MD5，≠ 测试集 |
| 坏例集 | 极端 / 历史问题样本 | 单独验收，不参与平均 |
| skip_layers | 跳过量化的层 | 混合精度，须写进 release |
| Release Note | 版本说明 | 锁工具链、驱动、验收数据 |
| PTQ | 后训练量化 | 训完再量化，靠校准集定 scale |
| Golden | PC 基准输出 | 张量对比的参考侧 |
