# GPU 工程师上 NPU，为什么编译过了还慢？

> **导读**：同样是 AI 加速芯片，GPU 和 NPU 的部署逻辑天差地别。GPU 卖通用算力和峰值吞吐；NPU 卖单位功耗下的确定性延迟——前提是你接受白名单、编译期优化和换芯片重来的约束。  
> **阅读时间**：约 20 分钟  
> **适合谁**：有 CUDA/TensorRT 经验、第一次上 NPU（昇腾/寒武纪等），或面试被问到「NPU 和 GPU 部署差在哪」的工程师。

---

## 决策卡点：选型会上的真实困境

某次边缘检测项目，YOLOv5s 在 RTX 3060 上单 batch 约 5ms，团队习惯 TensorRT 一把过。客户指定昇腾 310，ATC 编译也成功了，`.om` 能跑——但 msprof 一看，MAC 利用率只有 20% 出头，单帧 40ms，还偶尔抖一下。

老板问：「NPU 标称 16 TOPS，怎么比 GPU 还慢？」

问题往往不在「芯片假」，而在 **用 GPU 的思维做 NPU**：等着编译器自动优化、没做算子白名单检查、图里 Transpose 堆成串、动态 shape 硬上。NPU 和 GPU 从设计目标就不同——一个卖 **专用、低功耗、确定性**；一个卖 **通用、高吞吐、生态全**。部署链路里，NPU 把赌注压在 **编译期**；GPU 把赌注压在 **运行时**。搞清这一点，后面的 fusion、tiling、layout 才听得懂。

---

## 导读那句话怎么理解？

导读里的核心句是：

> GPU 卖通用算力和峰值吞吐；NPU 卖单位功耗下的确定性延迟——前提是你接受白名单、编译期优化和换芯片重来的约束。

不是在比「谁 TOPS 数字大」，而是在比 **各自擅长卖什么、你要不要为它付代价**。

### GPU 卖什么：通用算力 + 峰值吞吐

**通用算力**：GPU 不只为神经网络设计。训练、推理、科学计算、渲染都能干；CUDA + TensorRT 生态成熟，算子多、模型杂、改结构也快——「什么都能跑，大 batch 时往往还很快」。

**峰值吞吐**：云端 batch=32、64 批量推理时，GPU 很擅长把算力堆满，QPS 高。代价通常是 **功耗大**（百瓦级）、**单帧延迟有波动**（系统负载、调度会变）——边缘小 batch 不一定最划算。

打个比方：GPU 像 **瑞士军刀**——功能全、峰值猛，适合云端、大 batch、模型常改的项目。

### NPU 卖什么：单位功耗下的确定性延迟

**单位功耗**：同样跑一个检测模型，NPU 可能只要几瓦到十几瓦，GPU 常要上百瓦。无人机、摄像头、车载等 **电池/散热有限** 的场景，「每瓦能跑多少」往往比裸 TOPS 更重要。

**确定性延迟**：离线编译成 `.om` 后，同芯片、同输入，每次推理耗时通常很接近（波动常见在几个百分点内），P99 不会突然抖一大截。产线检测、自动驾驶感知等硬实时场景，要的不只是「平均 10ms」，还有「别偶尔变成 30ms」。

打个比方：NPU 像 **手术刀**——主要干神经网络推理这一件事，在低功耗 + 延迟稳定上往往更极致。

### 「前提是你接受约束」指什么？

NPU 的优势不是免费的，工程上要接受三件事：

| 约束 | 含义 | 工程感受 |
|------|------|----------|
| 白名单 | 只有厂商实现了的算子才在 NPU 上跑 | 模型里有特殊 `Softmax`、`GridSample` 等 → 要改图或 CPU fallback，性能可能崩 |
| 编译期优化 | 融合、tiling、layout 主要在 ATC 编译阶段定调 | 不能指望「扔给编译器自动变快」；改图 → 重编 → 常要数天 |
| 换芯片重来 | `.om` 绑定芯片型号，优化经验难照搬 | 昇腾 310 → 910，或换寒武纪，ONNX 可能复用，芯片模型和调参基本重做 |

不接受这些约束时，强行上 NPU 就容易变成：编译能过、profiler 利用率 20%、比 GPU 还慢——正是上文选型会里的困境。

### 合起来怎么选？

- 要 **通用、大 batch、快迭代、模型花样多** → 倾向 GPU（接受高功耗和延迟波动）。
- 要 **边缘、低功耗、小 batch、延迟要稳** → 倾向 NPU（接受白名单、编译期折腾、迁移成本高）。

不是「NPU 一定比 GPU 好或差」，而是 **你买的商品不同**。后文七大差异、全流程和选型表，都是在这句话底下展开。

**面试 30 秒口述**：GPU 是通用并行芯片，生态全，适合训练和大 batch 云推理，卖峰值吞吐。NPU 是专用神经网络芯片，优化在编译期，适合边缘低功耗，卖单位功耗下的确定性延迟。选 NPU 得接受算子白名单、编译期优化、换芯片重来的成本；选 GPU 得接受功耗和延迟波动。根据部署位置、功耗、batch 和迭代速度选，不是单纯比 TOPS。

## NPU 和 GPU 在部署链路里各干什么

```
  .pt (FP32)
      │
      ▼  torch.onnx.export + BN folding（NPU 侧常必须）
  ONNX (FP32)  ── ORT 验收：数学正确
      │
      ▼  算子支持性检查                    ← NPU：必做；GPU：多数可跳过
  ┌─────────────────────────────────────────────────────────────┐
  │ NPU：atc --check → 白名单 / fallback 标记                    │
  │ GPU：trtexec / ORT 直接转，unsupported 少见                   │
  └─────────────────────────────────────────────────────────────┘
      │
      ▼  PTQ / 校准（AMCT 等）→ INT8 ONNX（若需要）
      │
      ▼  编译 / 转换                         ← 最大分叉点
  ┌──────────────────────┬──────────────────────────────────────┐
  │ NPU：离线编译（ATC）   │ GPU：运行时编译（TensorRT 首帧等）     │
  │ 输出 .om，优化在编译期 │ 输出 engine，优化在运行时 + 编译期混合  │
  │ fusion/tiling/layout │ kernel 调度、显存、动态并行             │
  └──────────────────────┴──────────────────────────────────────┘
      │
      ▼  芯片模型 → 实机 profiling
  ┌─────────────────────────────────────────────────────────────┐
  │ NPU：msprof → MAC 利用率、DDR、算子级耗时                     │
  │ GPU：nsys / TRT profiler → kernel 耗时、吞吐、显存           │
  └─────────────────────────────────────────────────────────────┘
      │
      ▼  ACL / TensorRT 推理接口 → 上线
  交付
```

我在项目里习惯：**同一张 ONNX，NPU 和 GPU 各走一遍链路，对比的不是「谁 TOPS 大」，而是「谁在你的约束下更稳、更省、更快」**——约束包括功耗、batch、迭代频率、算子支持。

## 零基础先搞懂：本篇会反复出现的名词

下面按「直觉 → 部署里怎么用」写。后文再出现这些词，可以直接当复习。

### NPU 和 GPU：卖的不是同一样东西

**GPU（Graphics Processing Unit，图形处理器）** 最初为渲染设计，后来变成 **通用并行计算** 芯片：训练、推理、科学计算都能干，CUDA 生态极成熟。

**NPU（Neural Processing Unit，神经网络处理器）** 从第一天就为 **卷积、矩阵乘** 这类 NN 核心算子优化，其他操作是次要的。

打个比方：GPU 是瑞士军刀，什么都能做；NPU 是手术刀，只做神经网络推理，但在功耗和确定性延迟上往往更极致。面试别比「谁 TOPS 大」，要比 **你的场景买不买账**。

### 离线编译 vs 运行时编译

**离线编译**：在开发机上用 converter（如昇腾 **ATC**）把 ONNX 一次性编成芯片二进制（如 `.om`），优化（融合、tiling、layout）在编译阶段做完。编过以后，同型号芯片上推理路径通常很固定。

**运行时编译**：模型第一次推理时才编译成 engine（如 TensorRT），编译时间短，但优化深度和确定性不如 NPU 离线编译那么「死」——系统负载、驱动版本、输入 shape 变化都可能带来波动。

工程差异：NPU 改图 → 重新 ATC + AutoTune，周期常以 **天** 计；GPU 换 ONNX 常几小时内能验证，结构小改甚至不用重编 engine。

### ATC、CANN、ACL（昇腾侧）与 CUDA、TensorRT（GPU 侧）

**CANN** 是昇腾芯片软件栈（算子库、编译器、驱动）。**ATC（Ascend Tensor Compiler）** 把 ONNX 转成 `.om` 的 converter。**ACL（Ascend Computing Language）** 是推理时的 C++/Python API，管内存、设备上下文、加载 `.om`。

GPU 侧对应关系大致是：**CUDA** = 底层运行时，**TensorRT** = 推理优化与 engine 生成。换芯片时，ONNX 常能复用；**converter 参数、芯片模型、推理 API 代码** 往往要重做。

### 算子（Operator）与白名单

**算子** 是计算图里的节点：`Conv`、`Softmax`、`Transpose` 等。Netron 里每个方框基本对应一个算子。

NPU 采用 **白名单**：只有厂商实现了的算子才在 NPU 上跑；不在名单上的会 **CPU fallback（回退 CPU）**——日志里常有 `unsupported`、`CPU fallback`。GPU 生态开放，标准 ONNX 算子几乎都能跑，自定义算子用 CUDA 写也相对成熟。

NPU 部署 **第一步** 往往是算子支持性检查，不是直接开编译。

### 算子融合（Fusion）

**Fusion** 把多个算子合成一次执行，比如 `Conv + BN + ReLU`，省中间 feature 的 DDR 读写。

GPU 上 TensorRT 自动融合能力较强；NPU 上编译器也能融合，但 **很吃图结构**——中间插了 `Transpose`、`Cast` 就可能把链打断。工程师常要主动调算子顺序、做 BN folding，引导编译器融上。

### Tiling（分块）与 AutoTune

**Tiling** 把大张量切成小块，塞进片上 SRAM，减少反复读 DDR。**AutoTune** 在编译期搜索较优的分块参数。

Tiling 选差了，MAC 闲着、DDR 顶满，性能差一个数量级不稀奇。NPU 上这往往是 **核心优化手段**；GPU 上编译器多数自动处理，工程师干预较少。

### Layout：NCHW、NHWC、NC1HWC0

**Layout** 是数据在内存里的排布方式。**NCHW** 是 `[batch, channel, height, width]`，PyTorch 常用。**NC1HWC0** 是昇腾等 NPU 的原生布局，通道维会按硬件块（如 16）对齐。

非原生 layout 会触发 **Transpose**，一次 Transpose 就可能让整层慢很多。NPU 部署常要求 **导出阶段就少 Transpose**；GPU 编译器多数能自动转换。

### 静态 shape vs 动态 shape

**静态 shape**：输入尺寸编译时写死，如 `[1, 3, 640, 640]`。编译器能规划满内存，延迟抖动小。

**动态 shape**：batch 或高宽运行时变。GPU（TensorRT、ORT）支持较好；多数 NPU 偏静态，动态轴常走慢路径或需为每个尺寸单独编译。

车载、安防固定机位我默认静态；产品硬要多分辨率，先查 toolchain release note，再决定「一份动态模型」还是「三份固定尺寸模型」。

### TOPS、MAC 利用率与能效比

**TOPS（Tera Operations Per Second）** 是 datasheet 上的 **理论峰值算力**，不等于实机速度。**MAC 利用率** = 实际吞吐 ÷ 峰值，NPU 上编译期优化好坏直接决定这个数字；GPU 上运行时波动大，常见 30%～60%，优化好的 NPU 模型有时能到 60%～80%（视芯片和模型而定）。

**能效比** 是单位功耗下的算力。昇腾 310 典型功耗约 8W 量级，桌面 GPU 常 100W+——边缘电池供电场景，能效往往比峰值 TOPS 更重要。别迷信标称 32 TOPS：优化差的 32T 可能跑不过优化好的 16T。

### Batch size：延迟 vs 吞吐

**Batch size** 是一次推理喂多少张图。**Batch=1** 看单帧延迟，边缘实时常见；**Batch=32～128** 看吞吐，云端批处理常见。

NPU 在小 batch、低功耗下往往更有优势；GPU 在大 batch 峰值吞吐上通常更强。对比性能时 **batch 不对齐** 是最常见的公平性翻车原因之一。

### 确定性延迟

**确定性延迟** 指同样输入、同样环境下，每次推理耗时波动小。自动驾驶、产线检测、安防实时告警等场景，不只关心平均 ms，还关心 **P99 会不会突然抖一下**。

NPU 离线编译后，同 `.om` 同芯片上延迟通常较稳，波动常见在 5% 以内（视任务而定）；GPU 受调度、负载影响，波动 10%～30% 我见过不少。

### BN folding（批归一化折叠）

**BN（Batch Normalization）** 推理时可合并进 `Conv` 的权重，叫 **BN folding**。NPU 导出 ONNX 前 **常必须手动做完**；GPU 侧 TensorRT 多数能自动处理。没 fold 的 BN 会挡融合，多一次 DDR 读写。

### Profiler：msprof 与 nsys

**Profiler** 看实机谁慢、慢在哪。**msprof**（昇腾）导出算子级耗时、MAC 利用率、DDR 带宽；**nsys** / TensorRT profiler（英伟达）看 kernel 耗时、显存、调度。

NPU 性能问题 **只看总 latency 不够**，必须看 MAC 和 DDR——否则不知道是该改 tiling 还是改图。

## 设计目标不同：通用并行 vs 专用神经网络

所有部署差异的根源，是芯片 **设计目标** 不同。

GPU 要服务训练、推理、科学计算、渲染等 **通用并行** 任务，算子全、生态成熟，适合 **大 batch 云推理** 和快速迭代。NPU 专做 **神经网络推理**，固定图、低功耗、算子白名单，优化压在 **编译期**（fusion、tiling、layout）。换一颗 NPU，ONNX 往往还能用，但 **芯片模型和优化经验** 常要重做。

设计目标不同，是上文「导读那句话」的底层原因；下面按编译、算子、优化等维度，看工程里具体差在哪。

---

## 七大差异，工程里怎么感受

### 编译机制：离线 vs 运行时

NPU（以昇腾 310 为例）在 **ATC 阶段离线编译**，产出 `.om`；编译可能要几分钟到几小时（开 AutoTune 更久），但编通过后，同型号芯片上路径通常固定。GPU（TensorRT）多在 **首帧或 build 阶段** 编译 engine，几秒到几分钟，但运行时还会随负载、shape 微调调度。

| 维度 | NPU（昇腾 310） | GPU（英伟达） |
|------|-----------------|---------------|
| 编译时机 | 离线，ATC 一次性 | 运行时 / build 时 |
| 优化深度 | 融合、tiling、layout 多在编译期 | 编译期 + 运行时混合 |
| 确定性 | 同 `.om` 同芯片，延迟通常很稳 | 受负载、驱动影响，有波动 |
| 动态 shape | 多数偏弱，常需静态或多份模型 | 原生支持较好 |

表后要记：NPU 是 **「编译一次，同芯片到处跑」**；GPU 是 **「编译和运行、环境绑得更紧」**。NPU 迭代慢一轮，GPU 换 ONNX 往往更快。

### 算子：白名单 vs 开放生态

NPU 只有厂商实现的算子才在芯片上跑，unsupported 会 **CPU fallback**，一两个关键节点 fallback 就可能让延迟崩掉。GPU 标准算子覆盖极广，自定义用 CUDA 也相对成熟。

NPU 部署 **第一步** 应是 `atc --check` 或等价检查 + 对照团队算子知识库；GPU 多数项目可跳过这步。我见过最多的 NPU 首周翻车，是 **没做算子检查就开了 AutoTune**——编译能过，性能却像假 NPU。

### 优化：编译期静态 vs 运行时动态

GPU 工程师转 NPU 最不适应的，是 **不能再「扔给 TensorRT 自动优化」**。NPU 上性能多半在 **编译期** 定调：图结构是否利于融合、tiling 是否搜对、layout 是否原生。推理阶段可调的空间通常很小。

GPU 侧核心工作是 **选对引擎 + 调 workspace / batch**；NPU 侧核心工作是 **引导编译器**——改图、开 fusion 开关、跑 AutoTune、对 profiler 结果迭代。

### 性能：确定性低延迟 vs 峰值吞吐

同模型、同输入下（数字因环境而异，仅建立直觉）：昇腾 310 跑 YOLOv5s 单 batch 常见 ~10ms 量级且波动小；RTX 3060 单 batch 可能更快（~5ms），但波动 2～10ms 我见过不少。大 batch 吞吐则 GPU 往往更强。

| 场景 | NPU 更常见优势 | GPU 更常见优势 |
|------|---------------|---------------|
| 单 batch 延迟稳定性 | 波动小，适合硬实时 | 峰值可能更低，但抖动大 |
| 大 batch 吞吐 | 中等 | 高 |
| INT8 | 路径确定，常接近 FP16 的 ~2× | 好但受算子类型影响大 |

边缘端、嵌入式、**延迟波动要求严** 的场景，NPU 往往更合适；云端、**大 batch、要峰值 QPS** 的场景，GPU 往往更合适。

### 功耗与能效

昇腾 310 典型功耗约 **8W** 量级，被动散热、USB 供电可部署；桌面 GPU 常 **100W+**，要风扇和稳压电源。无人机、手持终端、摄像头等 **电池供电** 场景，能效比往往比裸 TOPS 更能决定选型。

### 生态与迁移成本

CUDA 生态发展二十年，资料、社区、工具链极成熟；CANN 等 NPU 栈国内资料渐多，但换 **芯片型号、CANN 版本** 时，算子表、融合规则、tiling 经验 **常不能照搬**。

| 可复用性 | NPU | GPU |
|----------|-----|-----|
| ONNX | 通常可以 | 可以 |
| 芯片模型（.om / engine） | 绑定芯片型号 | 绑定 GPU 架构 |
| 优化经验（tiling、融合） | 换芯片常重做 | 同架构间多数可迁移 |
| 推理代码 | ACL 等绑定厂商 | TensorRT 跨型号相对方便 |

换一颗 NPU，**几乎要重走一遍部署优化**——这是 NPU 项目预算里常被低估的成本。

## 部署全流程：同一张 ONNX，步骤差在哪

以昇腾 310 和英伟达 GPU 为参照，同一条 ONNX 走完全链路，阶段差异大致如下。

**模型导出与预处理。** NPU 侧常须在训练框架里 **手动 BN folding**，移除阻断融合的 `Transpose`/`Cast`，维度对齐到 16 的倍数；GPU 侧多数可交给编译器。INT8 量化：NPU 常用 **AMCT** 等厂商工具，与硬件绑定深；GPU 可用 TensorRT 内置或第三方，方式更灵活。

**算子兼容性。** NPU：**必须** `atc --check`、查知识库、准备替代子图；unsupported 不能留着，否则 fallback 性能断崖。GPU：标准模型几乎直接过。

**编译与参数。** NPU：离线 `atc` → `.om`，开 optimize、fusion、AutoTune，**指定静态 shape**；参数不对，性能差 3～5 倍我见过不少。GPU：`trtexec` 或 API build engine，支持动态 shape，workspace 给够即可。

**性能优化。** NPU：主动改图促融合、重跑 AutoTune、转 **NC1HWC0** layout、减 Transpose。GPU：调高 optimize level，多数自动完成。

**精度与性能验收。** NPU：建议逐层或至少关键层对比 FP32，量化/替换算子重点查；profiler 必看 **MAC、DDR**，不能只看总 ms。GPU：多数对比最终输出 + 任务指标即可；nsys 看 kernel 和显存。

**上线与维护。** NPU：`.om` 打包进应用，ACL 管内存；**每次改模型、升 CANN** 常要重编译 + 重测。GPU：换 ONNX 或 engine 相对快；升 CUDA 向下兼容通常更好。

| 阶段 | NPU 关键动作 | GPU 关键动作 |
|------|-------------|-------------|
| 导出 | BN fold、减 Transpose、对齐 16 | 多格式可进，预处理要求松 |
| 算子 | 白名单检查、改图 | 多数跳过 |
| 编译 | 离线 ATC + AutoTune + 静态 shape | TRT build，可动态 shape |
| 优化 | 融合/tiling/layout 人工引导 | 引擎自动为主 |
| 验收 | msprof：MAC、DDR、逐层精度 | profiler：kernel、吞吐 |
| 迭代 | 改模型 → 重编，周期常数天 | 换 ONNX，周期常数小时 |

## 项目里怎么选：边缘 vs 云端

| 场景 | 更倾向 NPU | 更倾向 GPU |
|------|----------|-----------|
| 部署位置 | 边缘、嵌入式、设备端 | 云端、数据中心 |
| 实时性 | 延迟 <50ms，波动希望 <5% | 延迟 <500ms 可接受，波动 10%～30% 可接受 |
| 功耗 | 严格（如 <20W） | 宽松（>100W 可接受） |
| Batch | 小（1～4） | 大（32～128） |
| 模型 | 主流 CV（YOLO、ResNet） | 大模型、NLP、快速试错 |
| 迭代 | 结构相对固定 | 频繁改结构 |

**决策链（口述版）：**

- 部署在边缘 / 嵌入式？→ 优先看 NPU  
- 延迟确定性要求高（波动希望很小）？→ 优先 NPU  
- 功耗预算很紧？→ 优先 NPU  
- 模型每周改结构、要快迭代？→ 优先 GPU  
- 大模型 / 复杂 NLP？→ 优先 GPU  
- 以上都不极端 → 按团队栈和已有硬件定

NPU 和 GPU 不是谁替代谁，是 **互补**。部署工程师的工作是根据业务约束选对芯片，并把它的性能榨出来。

## 最常见坑（附排查顺序）

### 坑 1：用 GPU 思维等编译器自动优化

**典型做法**：ONNX 直接扔 ATC，不开 fusion、不改图、不跑 AutoTune。

**现象**：编译通过，MAC 利用率 20% 以下，比 GPU 还慢。

**原因**：NPU 性能多半在编译期定调，编译器不会替你改图。

**怎么查**：msprof 看 MAC/DDR；dump 图看融合是否生效。

**怎么改**：BN fold、减 Transpose、开 AutoTune、按 profiler 迭代 tiling。

### 坑 2：跳过算子白名单检查

**典型做法**：ORT 能跑就直接 ATC，不做 `atc --check`。

**现象**：能推理，但延迟莫名高；日志有 `CPU fallback`。

**原因**：关键算子在 CPU 上跑，NPU 空转。

**怎么查**：编译日志 + profiler 的 Device 列。

**怎么改**：等价子图替换、拆算子、或登记自定义算子；结论写入算子知识库。

### 坑 3：迷信 TOPS 标称值

**典型做法**：选型只看 datasheet 32 TOPS，不看优化和利用率。

**现象**：实机远不如预期，和「16T 竞品」比反而输。

**原因**：TOPS 是峰值；实际 = 峰值 × MAC 利用率 × 有效算子占比。

**怎么查**：msprof MAC 利用率；对比同模型优化前后。

**怎么改**：把预算花在编译期优化；选型时要求 **同模型 benchmark**，不要只比 TOPS。

### 坑 4：动态 shape 硬上 NPU

**典型做法**：训练图带动态 batch/分辨率，不做静态化直接编译。

**现象**：首帧慢、尺寸一变就抖、或每尺寸重新编译。

**原因**：NPU 对动态轴支持边界窄，常走慢路径。

**怎么查**：日志 dynamic shape 警告；多尺寸各测 latency 分布。

**怎么改**：能静态就静态；必须多尺寸则 **多份固定 shape 模型** 或查 toolchain 对动态 INT8 的支持说明。

## 面试追问

### 1

**问：** NPU 和 GPU 设计目标差在哪？部署上最直观的一点是什么？

**答：** GPU 为通用并行计算设计，卖通用算力和峰值吞吐，适合训练和大 batch 云推理。NPU 为专用神经网络设计，卖单位能耗下的确定性延迟，适合边缘和硬实时。部署上最直观的是 **优化发生在哪**：NPU 主要在 **编译期**（ATC 出 `.om`），GPU 主要在 **运行时**（TensorRT 调度 + 首帧编译）。

### 2

**问：** 什么叫离线编译？和 TensorRT 运行时编译有什么工程差异？

**答：** 离线编译是在开发机上用 ATC 等工具把 ONNX 一次编成 `.om`，融合、tiling、layout 在编译阶段做完，同芯片上推理路径通常很固定。TensorRT 多在 build 或首帧生成 engine，周期短，但受驱动、负载、shape 影响，延迟波动更大。工程上 NPU 改模型常要 **重编 + AutoTune，数天**；GPU 换 ONNX 常 **数小时** 能验证。

### 3

**问：** NPU 白名单机制是什么意思？unsupported 了你怎么处理？

**答：** 只有厂商实现了的算子才在 NPU 上跑，不在名单上的会 CPU fallback，关键路径 fallback 延迟会崩。我会先 `atc --check` 和 Netron 定位节点，查算子表和团队知识库，优先 **数学等价子图替换**，其次拆算子，最后才考虑自定义 NPU 算子。改完 ORT 对齐输出，再编译；通过后记进知识库。

### 4

**问：** 边缘端实时检测，为什么有时选 NPU 不选 GPU？

**答：** 边缘常要 **低功耗**（电池供电）、**小 batch 稳定延迟**（波动希望小）、**固定机位静态 shape**。NPU 离线编译后同 `.om` 上延迟通常较稳，8W 量级能跑；GPU 峰值可能更低，但 100W+ 功耗和 10%～30% 波动在产线/车载里有时不可接受。当然，若模型 NLP 算子多、迭代快，或团队只有 CUDA 栈，GPU 或边缘 GPU 也可能更合适。

### 5

**问：** 换一颗 NPU 芯片，哪些能复用、哪些要重做？

**答：** **ONNX 和改图经验** 多数能复用；**`.om` 芯片模型、tiling/融合参数、ACL 集成代码、profiler 调优结论** 常要重做。不同厂商、不同 CANN 版本算子表和融合规则差很多，和 GPU 同架构间迁移 engine 比，NPU **跨芯片成本通常更高**。项目排期要把 **完整部署优化流程** 算进去，别只估 converter 转换时间。

## 术语速查

| 术语 | 全称 / 含义 | 一句话直觉 |
|------|------------|-----------|
| NPU / GPU | 神经网络处理器 / 图形处理器 | 手术刀 vs 瑞士军刀；卖的东西不同 |
| 离线编译 | ATC → `.om` | 优化在编译期；改图要重编 |
| 白名单 / fallback | — | 不支持的算子回 CPU，延迟易爆 |
| Fusion | 算子融合 | Conv+BN+ReLU 合并；NPU 要引导 |
| Tiling | 分块 | 切大块 fit SRAM；AutoTune 搜参数 |
| NC1HWC0 | 昇腾原生 layout | 非原生要多 Transpose |
| 静态 / 动态 shape | — | NPU 偏静态；GPU 动态友好 |
| MAC 利用率 | — | 实机算力吃满几成；别只看 TOPS |
| AMCT | 昇腾量化工具 | INT8 与硬件绑定，和 TRT 量化不通用 |
| msprof / nsys | profiler | NPU 看 MAC/DDR；GPU 看 kernel |
