# 客户要动态多分辨率，静态 640 不想用——NPU 上怎么选？

> **导读**：「支持任意分辨率」是边缘检测项目的高频需求，但 NPU 的性能优势来自编译期静态优化，动态 shape 和最优性能天然打架。工程上往往不是「能不能做全动态」，而是「代价多大、值不值」——多数项目最后落在 2～3 档静态模型，而不是一份万能动态模型。  
> **阅读时间**：约 22 分钟  
> **适合谁**：ORT 能跑、客户嫌固定 640 太死板、要在 NPU 上落地多分辨率需求的部署工程师。

---

## 决策卡点：选型会上的真实困境

某次安防边缘项目，客户口头说「320 到 1280 都要支持」，明确不想用固定 640×640 的静态模型——「画面远近切换，小目标看不清」。团队按 CANN 文档开了 HW 动态编译，ATC 编过，640 上延迟 18ms，看着还行。

上线第一周就出事：夜班切到 1280×720 源流，单帧延迟飙到 45ms，P99 抖动超过产线红线；更糟的是，同一模型在 320×320 下 mAP 掉了近 3 个点，框位置还偶发整体偏移。复盘才发现三件事：动态模式下 AutoTune 没生效、两个卷积层静默 fallback 到 CPU、校准集只有 640 尺寸——开发阶段只验了「常用档」，没验最坏档。

这类坑我见得多的，根因往往不是「芯片不支持动态」，而是 **用 GPU 的动态思维直接套 NPU**：没先收窄需求、没查工具链边界、没拿最大分辨率当验收标准。下文按「先搞懂矛盾 → 问清需求 → 查工具链 → 选方案 → 避坑验收」展开，新人读完应能讲清四种落地方案怎么选、面试被追问时怎么答。

## 动态多分辨率在部署链路里干什么

```
  业务需求：多分辨率输入（320～1280、横竖屏、远近变焦……）
      │
      ▼  需求收窄（本文第一节）          ← 很多「全动态」其实只需 2～3 档
      │
      ▼  导出 ONNX
  ┌─────────────────────────────────────────────────────────────┐
  │  static shape（无 dynamic_axes）  vs  dynamic_axes（H/W 可变）│
  │  边缘 NPU 默认倾向：静态；动态要先查 toolchain release note   │
  └─────────────────────────────────────────────────────────────┘
      │
      ▼  PTQ / 校准（若 INT8）
  ┌─────────────────────────────────────────────────────────────┐
  │  校准集须覆盖目标分辨率；单尺寸校准 → 其他尺寸精度波动大        │
  └─────────────────────────────────────────────────────────────┘
      │
      ▼  编译策略（本文核心分叉）        ← 多份静态 om / letterbox / 原生动态
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │ 方案一       │ 方案二       │ 方案三       │ 方案四       │
  │ 多档静态模型 │ letterbox    │ 工具链动态   │ 云边协同     │
  │ 2～3 份 .om  │ 统一到固定HW │ 单份动态 om  │ 边缘小+云端大│
  └──────────────┴──────────────┴──────────────┴──────────────┘
      │
      ▼  预处理 + 后处理
  ┌─────────────────────────────────────────────────────────────┐
  │  letterbox 缩放/填充须与训练一致；各档坐标逆变换单独验证       │
  └─────────────────────────────────────────────────────────────┘
      │
      ▼  实机验收：最坏分辨率下的延迟、内存、精度、稳定性
  交付
```

一句话：**分辨率策略在「导出 shape 定调」和「编译出几份芯片模型」两处做决定**；后处理坐标映射错，张量再对齐框也会全乱——这和量化无关，是预处理/逆变换没跟分辨率档位对齐。

## 零基础先搞懂：本篇会反复出现的名词

### NCHW 和 `[1, 3, 640, 640]` 各维什么意思

`NCHW` 是深度学习里常见的四维张量排布：`N` batch（批大小）、`C` channel（通道，RGB 为 3）、`H` height（高）、`W` width（宽）。`[1, 3, 640, 640]` 即单张图、三通道、640×640 像素。

导出 ONNX 时若 **不写** `dynamic_axes`，四维数字在图里是固定的，这叫 **静态 shape**。若写了 `dynamic_axes` 把 `H`、`W` 或 `batch` 标成可变，图里对应维可能是 `-1`，这叫 **动态 shape**。注意：**业务上用 160×320 等非正方形输入**（像素总数可能比 640×640 少很多）和「导出动态轴」不是一回事——前者是固定 shape 换尺寸，后者是编译器运行时接受多种尺寸。

### 静态 shape vs 动态 shape（HW 动态）

**静态 shape**：编译时输入高宽写死。NPU 编译器能据此做满 **Tiling（分块）**、算子融合、内存地址规划，同 `.om` 上延迟通常稳定、抖动小。车载、安防固定机位，我默认静态，除非产品硬要多分辨率。

**动态 shape（尤其 HW 动态）**：高宽在运行时变。编译器无法为每一种尺寸都算出最优解，往往用 **保守 tiling**（按最大尺寸留余量）、关闭部分融合、甚至部分算子 **fallback 到 CPU**。能编过不代表跑得漂亮——面试别只说「动态更灵活」，要说你会先翻 toolchain release note，再决定「一份动态模型」还是「三份固定尺寸模型」。

### Tiling、融合、AutoTune（为什么动态会慢）

**Tiling**：把大张量切成小块放进芯片片上 SRAM（类似 CPU 的 L1/L2），减少反复读 DDR。**算子融合**：把 Conv+BN+ReLU 等合并成更少 kernel，少访存。**AutoTune**：编译器自动搜索 tiling 等参数。这三样都依赖 **已知的输入 shape**。分辨率一变，优化空间塌缩——这是 NPU「不待见动态」的底层原因，不是某一家厂商故意刁难。

### letterbox（等比缩放 + 补边）

保持原图宽高比，缩放到能放进目标框（如 640×640），不足部分用灰边（常见 114,114,114）填充。训练、推理、后处理 **必须用同一套规则**；推理后还要做 **letterbox 逆变换** 把框坐标映射回原图。若端侧用 stretch 硬拉成正方形，却按 letterbox 公式逆变换，框必偏——这和模型、量化无关。

letterbox 里的 `scale` 是「原图缩到网络输入」的比例，和 INT8 量化里的 `scale`（浮点映到整数的系数）是 **两个概念**，别混。

## 为什么 NPU 和「全动态分辨率」天然别扭？

NPU 的性能优势，核心在 **编译期静态优化**：Tiling 分块、算子融合、内存规划、指令调度，都基于 **固定的输入 shape** 算出来的。一旦高宽可变，编译器只能走保守路径：

1. Tiling 按 **最大尺寸** 留余量，小图时 SRAM 利用率掉下去，算力空转。
2. 部分融合在动态 HW 下会关掉，避免融合后 shape 对不上。
3. 个别算子（大 kernel 卷积、特殊归一化等）动态 HW 不支持，**静默 fallback CPU**，profiler 不细看像「编过了但莫名慢」。
4. 内存按最大尺寸预留，小分辨率也占着大块 buffer，边缘设备容易 OOM。

动态分辨率换的是 **输入灵活性**，牺牲的是 **性能、功耗、延迟确定性和可维护性**。没有免费的灵活——这和 GPU 上 TensorRT 动态 shape 相对友好，是两条不同的产品路线。

## 动手前：先把需求问透

多数「要任意分辨率」的需求，对齐四个问题后会收窄很多；我见过的项目里，真正需要连续全动态的不到两成。

**1. 分辨率范围**：最小、最大各多少？是 **连续变化** 还是 **固定几档**？客户说「320～1280 都要」，实际业务可能只用 320、640、1280 三档——档位摸清，方案往往直接落到多份静态模型。

**2. 切换频率**：每帧都可能变，还是 **启动时选一次** 就不变？启动选一次可以启动时预加载全部档位，运行时切换上下文，延迟常在毫秒级以内；每帧都变要算模型切换和预处理切换的开销。

**3. 延迟红线**：各分辨率下 P99 上限是多少？若要求和静态 640 **一样快**，全动态 HW 在多数 NPU 上很难达标——最坏档延迟往往是常用档的 1.5～2 倍，视芯片和图结构而定。

**4. 精度要求**：小分辨率下目标变小，mAP 掉 1～2 个点不罕见；客户是否接受？INT8 时不同尺寸的量化误差也不同，要在需求里写清 **按档验收** 还是 **全范围一个指标**。

对齐完，绝大多数场景会落在「2～3 档固定分辨率」区间，根本不需要「任意 HW」。

## 第一步：查工具链的动态能力边界

动手编译前，先查 **目标芯片 + 工具链版本** 的 release note 和算子支持表，不要想当然。下面以 **昇腾 310 + CANN** 为例说明思路（其他 NPU 对照自家文档，结构类似）。

### 动态模式大致分三档

| 动态类型 | 支持情况（昇腾生态常见） | 性能损耗（经验区间） | 工程感受 |
|---------|------------------------|---------------------|----------|
| 动态 Batch | 各版本普遍支持 | 约 10%～20% | 多路并发时常用，相对省心 |
| 动态分辨率（HW 动态） | CANN 5.1+ 需指定范围 | 约 20%～50%，最坏档可能更多 | 能编过 ≠ 全尺寸都快 |
| 任意维度全动态 | 支持边界窄 | 50% 以上或不可用 | 边缘项目一般不碰 |

表后补一句：**动态 Batch 和 HW 动态是两件不同的事**——客户说「要多分辨率」时，先确认他要的是改 H/W，还是改 batch；别混在一个编译参数里。

### 核心限制（动手前心里有数）

- 须在编译时指定 **分辨率范围**（最小 HW～最大 HW），不是数学上「任意正整数」。
- 超出范围的输入会直接报错，不能指望运行时自动缩放。
- 动态模式下 **AutoTune 常失效**，tiling 走通用保守策略。
- 部分算子动态 HW 不支持 → CPU fallback；编译日志不一定显眼，要配合 profiler 查 Device 列。
- **INT8 量化 + 动态 HW**：精度波动通常比 FP16/FP32 更大，校准难度高。

### 若坚持原生动态：必测最坏档

决定走工具链动态方案时，**验收必须以最大分辨率的延迟、内存、精度为准**——很多人只测 640 觉得 18ms 还行，上线 1280 直接翻倍或 OOM。同时测 **最小档**，看小目标是否精度异常、框是否整体偏移（常和后处理/letterbox 有关）。

## 四种落地方案：怎么选、怎么做

按 **工程稳定性从高、灵活性从低** 排序。工业落地优先前面的；后面的灵活，但排查成本和性能代价更高。

### 方案一：多档静态模型（多数项目的首选）

**做法**：按业务选 2～3 档核心分辨率（如 320、640、960），每档单独导出静态 ONNX、单独 ATC 编译成最优 `.om`；运行时根据输入（或产品配置）选最近档位，必要时 letterbox 到该档固定尺寸再推理。

**优点**：每档都是完整编译优化，性能和纯静态 640 一致；每档延迟、内存、精度 **可预期**，问题定位和静态流程相同；2～3 份模型合计往往也就几十 MB，边缘存储可接受。

**缺点**：多占一点存储；档位切换若每次从磁盘加载模型，可能有几百毫秒卡顿——**启动时预加载全部档位到内存**，运行时只切换推理上下文，切换延迟通常可压到 1ms 量级（视 ACL 用法而定）。

**适用**：常用档位 ≤3、延迟和稳定性要求高、生产环境要可排查。我见过的工业项目，最后落在这档的最多——看似多了模型，总维护成本往往 **低于** 一份全动态 om。

**落地细节**：

- **档位间隔**：相邻档约 1.5～2 倍，如 320 / 640 / 960，覆盖远近景又不至于模型太多。
- **就近适配**：输入落在两档之间时，常选 **就近高一档**（保留细节），再 letterbox 到该档网络输入。
- **版本管理**：每档 om 与对应 ONNX、校准集 hash、工具链版本一起写进 release note，半年后能复现。

### 方案二：统一 letterbox 到固定分辨率（最简）

**做法**：不管输入多大，预处理一律 letterbox 到固定尺寸（如 640×640），**只维护一份**静态模型。

**优点**：一份模型、一套编译和量化流程，维护成本最低；计算量固定，延迟完全确定；只改预处理，不动 ATC 参数，周期紧时很香。

**缺点**：小图 letterbox 后大量灰边 padding，**无效算力**多；极端小尺寸下目标缩得太小，mAP 可能掉点。分辨率跨度大（如 320 和 1280 同源）时，统一到 640 往往不如分档静态划算。

**适用**：分辨率范围不大（如 480～640）、最大尺寸算力可接受、不想维护多版本、对成本敏感。

**注意**：letterbox 填充色、插值算法（双线性/最近邻）须与 **训练完全一致**；后处理逆变换的 `scale`、`pad_w`、`pad_h` 要按 **实际 letterbox 结果** 算，不能共用一套参数糊弄多档输入。

### 方案三：工具链原生动态分辨率（灵活但有代价）

**做法**：用厂商动态 HW 编译选项，生成 **单份** 支持指定范围内的模型；运行时传入实际 H、W。昇腾 ATC 示例（尺寸列表按业务改）：

```bash
atc --model=model.onnx \
    --output=model_dynamic.om \
    --framework=5 \
    --input_format=NCHW \
    --input_shape="images:1,3,-1,-1" \
    --dynamic_image_size="images:1,3,320,320;1,3,640,640;1,3,960,960" \
    --soc_version=Ascend310 \
    --optimize=2
```

运行时通过 ACL 等接口设置本次推理的实际图像尺寸（须落在编译声明范围内）。**具体 API 以当前 CANN 文档为准**——不同版本字段名可能略有差异，编不过先对 release note。

**优点**：单模型，范围内不必为每档单独 letterbox 到不同网络输入（仍可能要预处理归一化）；理论上少一次「人为选档」的逻辑。

**缺点**：同尺寸下通常比静态 om 慢 20%～50%，越偏离编译器「偏好」尺寸越慢；算子 fallback、量化精度按尺寸波动，**问题难复现**；排查要像测多个静态模型一样，对 **范围内每个关键尺寸** 各测一遍。

**适用**：档位多且连续、无法用 2～3 档覆盖；延迟要求宽松；测试资源够铺全尺寸矩阵。

### 方案四：云边协同（架构级折中）

**做法**：边缘端固定小分辨率静态模型做实时检测、告警；少数需要高清复核的场景，原图上传云端，用大分辨率或大模型精修。

**优点**：边缘保持最优功耗和延迟；云端满足高清和多分辨率；整体成本可按流量折中。

**缺点**：依赖网络，纯离线不可用；端云协议、鉴权、带宽和时延要单独设计，架构复杂度高。

**适用**：大部分监控场景小图够用，少数异常需人工或云端复核，且网络稳定。

### 四方案一眼对比

| 方案 | 模型份数 | 性能 | 维护/排查 | 灵活性 | 我什么时候推 |
|------|---------|------|-----------|--------|-------------|
| 多档静态 | 2～3 | 最优 | 低 | 中（离散档） | 默认首选 |
| letterbox 固定 | 1 | 稳定 | 最低 | 低（逻辑统一） | 范围小、周期紧 |
| 原生动态 | 1 | 降 20%～50% 常见 | 高 | 高（范围内连续） | 档多、延迟不紧、测得过来 |
| 云边协同 | 边缘 1+ 云端 | 边缘最优 | 架构高 | 业务分层 | 大流+偶发高清 |

## 选型怎么走：决策顺序

不必背树，按下面顺序问即可：

```
客户要动态多分辨率
    │
    ▼  对齐需求：实际是不是只有 2～3 档常用尺寸？
    │      是 → 【多档静态模型】（方案一）
    │      否、且范围大又连续 → 继续
    ▼
    性能：能否接受比静态同尺寸慢 20% 以上？
    │      不能 → 【letterbox 到固定尺寸】（方案二）
    │      能 → 继续
    ▼
    工具链：HW 动态支持度、最坏尺寸延迟/内存、有无 CPU fallback
    │      不满足 → 退回方案一或二
    │      满足 → 【原生动态】（方案三）
    ▼
    有网、少数高清复核需求 → 可叠加【云边协同】（方案四）
```

面试口述可以压成三句：**先收窄档位 → 再查 toolchain 最坏档 → 默认 2～3 份静态 om，全动态是少数且要付性能和维护税**。

## 最常见坑（附排查顺序）

### 坑 1：只测常用尺寸，不测最坏档

**典型做法**：开发只在 640×640 上验延迟和 mAP，认为「动态模型搞定了」。

**现象**：上线最大分辨率延迟翻倍、内存 OOM，或夜班一切源流就超时。

**原因**：动态编译按范围预留资源，最坏档计算量和内存都是验收红线；常用档好看不代表全范围合格。

**怎么查**：用 **最大 HW** 跑满预热后测 P50/P99 延迟和峰值内存；profiler 看是否 DDR 打满。

**怎么改**：验收标准写入需求文档——**以最大分辨率为门禁**；不满足则改方案（多档静态或 letterbox），别硬上线。

### 坑 2：动态模式下算子静默 fallback

**典型做法**：ATC 编译无报错，直接上板。

**现象**：延迟比预期高很多，MAC 利用率低，但图「看起来」没问题。

**原因**：部分层在动态 HW 下不支持，走 CPU；编译日志不一定一眼可见。

**怎么查**：编译日志搜 `CPU`、`fallback`；msprof 等工具看算子 Device 列、耗时 Top。

**怎么改**：改图换等价算子、拆层，或退回静态多档；结论记入算子知识库。

### 坑 3：量化校准只有单分辨率

**典型做法**：INT8 校准集全是 640×640，其他尺寸直接上线。

**现象**：640 精度正常，320 或 960 mAP 明显掉，甚至框系统性偏移。

**原因**：激活分布随分辨率变，单尺寸定的 scale 在其他 HW 上不准。

**怎么查**：分尺寸跑同一测试集，对比 FP32 掉点；看 per-layer 余弦是否某档全面变差。

**怎么改**：校准集 **覆盖所有目标分辨率**；head 等敏感层考虑 skip quant 或混合精度；动态 HW + INT8 要更保守地验收。

### 坑 4：多档模型切换卡顿

**典型做法**：运行时按需从磁盘 `load` 模型。

**现象**：一切分辨率就卡几百毫秒，产线节拍乱。

**原因**：模型加载 IO 和解密开销大，不是推理本身慢。

**怎么查**：打时间戳区分 load 与 `execute`；看是否每次切换都 reload。

**怎么改**：启动时 **预加载全部档位** 到内存，运行时只换 context / 输入描述符；档位切换纳入性能测试用例。

### 坑 5：后处理坐标映射各档共用一套参数

**典型做法**：多档静态或 letterbox 方案里，C++ 后处理写死一组 `scale`、pad，或逆变换公式和 Python 训练不一致。

**现象**：张量余弦很高，检测框位置全乱或整体偏移。

**原因**：不同档位的 letterbox `scale`、`pad_w`、`pad_h` 不同，共用参数必错。

**怎么查**：同一网络输出分别用 Python 与 C++ 后处理比框；固定一张图逐档验逆变换。

**怎么改**：每档（或每次 letterbox 结果） **单独传** 逆变换参数。解码后在 letterbox 坐标系得到框 `(x, y, w, h)`，映射回原图时：`x_orig = (x - pad_w) / scale`，`y_orig` 同理，宽高也要除 `scale`；漏减 pad 或漏除 scale，张量再对齐框也会全飘。验收时对每档固定一张图，Python 与 C++ 后处理比数值，不要只肉眼看图。

## 验收标准怎么定

没有银弹，我会和客户对齐一张表（数值最终以业务线为准）：

| 维度 | 建议验收项 |
|------|-----------|
| 性能 | 各档位（或范围内最大 HW）P99 延迟、峰值内存、功耗 |
| 精度 | 相对 FP32 的 mAP/召回/误检掉点上限；分档报，不只看平均 |
| 稳定性 | 长跑 + 随机切档；无 OOM、无累积泄漏 |
| 可维护性 | 模型份数、切换逻辑、release note 是否可半年复现 |

动态方案 **必须** 在需求里写明：验收以 **最坏分辨率** 为准，不是「代表尺寸」。

## 面试追问

### 1

**问：** 客户不要静态 640，要动态多分辨率，你第一步做什么？

**答：** 先收窄需求，不是先开 dynamic_axes。问清实际常用几档、最小最大 HW、切换频率和延迟红线。然后查 toolchain 对 HW 动态和 INT8 的支持边界。不支持或最坏档不达标，就在 **多份固定尺寸模型、letterbox 到统一尺寸、云边分工** 里选；支持也要用 **最大分辨率** 验延迟和内存，很多项目最后折成 2～3 档静态 om，维护成本反而更低。

### 2

**问：** 静态 shape 和动态 HW，对 NPU 编译优化差在哪？

**答：** 静态 shape 下编译器能做满 tiling、融合和内存规划，延迟稳。动态 HW 只能保守 tiling、部分融合关闭，算子还可能 fallback CPU，同图同芯片常慢 20%～50%，视尺寸和图而定。所以 NPU 边缘默认静态；要多分辨率，优先 **多份静态模型** 而不是一份全动态。

### 3

**问：** 多档静态模型和 letterbox 到 640，怎么选？

**答：** 档位少、每档都要最优延迟和精度 → 多档静态，每档单独编译。分辨率跨度不大、想一份模型省事、能接受小图 padding 浪费算力 → letterbox 统一到 640。跨度大（320 和 1280 同源）时，letterbox 到中间尺寸往往不如分档；周期紧、范围小 letterbox 最快。

### 4

**问：** 动态分辨率模型 640 上精度正常，320 掉了，你会先查什么？

**答：** 若 INT8，先查校准集是否只有 640——这是最常见原因，补多分辨率校准再验。若 FP32 也掉，查小图目标尺度和 letterbox 后处理逆变换是否和训练一致。若只有动态 om 的某些尺寸慢，再查 profiler 有没有 CPU fallback 和该尺寸下的 tiling。别一上来改网络结构。

### 5

**问：** 导出 ONNX 时写了 dynamic_axes，是不是就能任意分辨率上 NPU？

**答：** 不是。ONNX 动态只说明图在数学上接受可变维；NPU 还要编译期声明范围，且很多算子动态 HW 支持不完整。能编过不等于全尺寸都快、都准。我会 ORT 先验数学正确，再 ATC 按 **业务尺寸列表** 编译，每档实测延迟和精度；面试里要体现 **toolchain 边界** 和 **最坏档验收**，不是「导出成功就算交付」。

## 术语速查

| 术语 | 全称 / 含义 | 一句话直觉 |
|------|------------|-----------|
| 静态 shape | 编译时 H、W 固定 | NPU 最爱；延迟稳、优化满 |
| 动态 HW | 运行时高宽可变 | 灵活但慢；须声明范围 |
| dynamic_axes | ONNX 导出选项 | 图可变 ≠ 芯片跑得漂亮 |
| Tiling | 分块 fit SRAM | 依赖固定 shape；动态则保守 |
| letterbox | 等比缩放+补边 | 预处理与逆变换必须成对 |
| fallback | 算子回退 CPU | 动态 HW 下更要查 profiler |
| AutoTune | 编译器搜参 | 动态模式常失效 |
| 多档静态 om | 每档一份芯片模型 | 工业界最常见折中 |
| ATC | Ascend Tensor Compiler | 昇腾 ONNX→`.om`；动态用 dynamic_image_size |
| P99 延迟 | 99 分位耗时 | 动态/切档验收看 P99，不只平均 |
