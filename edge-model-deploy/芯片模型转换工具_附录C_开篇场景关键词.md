## 先把整段话串起来

原文：

> 某次检测项目，新人把 `yolov5s.onnx` 交给 **ATC**，十分钟后拿到 `yolov5s.om`，以为部署完成了。上板 **msprof** 一看：单帧 45ms，**MAC 利用率**不到 25%；更糟的是，和 PC 侧 **ORT** 比精度，**mAP** 掉了 8 个点——团队先怀疑**校准集**，换了三版 **calib** 没变化。  
> 翻编译日志才发现：**Resize** 走了近似 nearest、**GridSample** 标记了 **CPU fallback**，中间还插了四个 **Transpose** 把 **Conv+BN+ReLU 融合链**全打断了。这些问题不是推理 **runtime** 能修的，它们在**编译**那一步就已经写进 `.om` 了。

用大白话复述：**人以为「格式转好了 = 部署好了」，结果板子上又慢又准；瞎调校准集没用，根子在编译日志里——有些算子算错了路、有些跑去了 CPU、该合并的计算没合并成。这些在生成 `.om` 时就定死了，上线后改不了。**

下面按出现顺序，逐个拆开。

## 文件与工具

### yolov5s.onnx

**是什么**：一个叫 YOLOv5s 的目标检测模型，存成 **ONNX 格式** 的文件。ONNX 可以理解成一份「计算说明书」：里面画了网络有哪些层（卷号、激活、拼接等），权重附在后面，**不绑定** 昇腾、英伟达或任何一块具体芯片。

**打个比方**：像建筑 **设计图纸**——说明要盖几层、多大面积，但还没指定用哪家施工队、用什么钢筋规格。

**在这句话里**：新人手里有的是「通用图纸」，还没变成「昇腾施工方案」。

---

### ATC（Ascend Tensor Compiler，昇腾张量编译器）

**是什么**：华为昇腾芯片配套的 **离线编译工具**。你把 ONNX 丢给它，它做算子检查、图优化、分块、布局转换等，最后吐出 `.om` 文件。

**打个比方**：图纸交给 **施工队总包**——他们决定哪些工序用自有设备干、哪些外包给杂工（CPU），材料怎么码放（Layout），工地怎么分段施工（Tiling）。

**在这句话里**：新人只做了「把 onnx 交给 ATC 编一下」，没看编出来的 **施工记录（编译 log）**。

---

### yolov5s.om

**是什么**：ATC 编译后的 **昇腾离线模型**。里面已经是芯片能直接执行的指令、排好序的权重、内存地址规划等，推理程序加载它就能跑，**不再读 ONNX**。

**打个比方**：不是图纸了，是 **竣工图 + 施工手册 + 预制件清单** 打包成一个文件，专给这块昇腾板子用。

**在这句话里**：新人以为拿到 `.om` 就等于「部署完成」——其实只完成了 **编译**，后面还有上板、测速、测精度、查 log。

---

## 性能相关

### msprof

**是什么**：昇腾上的 **性能分析工具**（MindStudio 里常用）。能在实机跑推理时统计：每层耗时、MAC 用了多少、DDR 搬了多少数据、有没有算子跑在 CPU 上等。

**打个比方**：给工厂装 **监控摄像头 + 工时表**——不是看产品对不对，是看流水线哪一段在摸鱼、哪条线在等料。

**在这句话里**：上板后一 profiling，发现整体慢、算力没吃满，说明 **编译期定的执行路径有问题**，不是简单调个 batch 能救的。

---

### 单帧 45ms

**是什么**：处理 **一张图**（一帧输入）从开始推理到出结果，花了约 45 毫秒。检测项目里常用来估 **延迟**：45ms 大约每秒 22 帧（1000÷45），是否满足实时要看产品要求。

**打个比方**：做一道菜从下单到上桌 45 分钟——客户要 15 分钟上菜，那就是不达标。

**在这句话里**：编过了、能跑，但 **速度不合格**；后面 MAC 利用率低解释了「为什么慢」。

---

### MAC 利用率（不到 25%）

**是什么**：NPU 里负责矩阵乘、卷积的 **MAC 单元**（Multiply-Accumulate，乘加运算核心）实际干活的时间占比。标称 16 TOPS 是峰值能力；利用率 25% 意味着 **大部分时间在等数据、跑杂活、或在 CPU 上算**，没把算力芯吃满。

**打个比方**：餐厅有 10 个灶台（MAC），高峰只开了 2～3 个在炒菜，其余在等菜切好送过来（DDR 带宽、CPU fallback、碎 kernel 太多）。

**在这句话里**：和 45ms 对应——**不是芯片坏了，是编译出来的执行方式让算力空转**。后面 Transpose 多、融合断、GridSample 跑 CPU，都会拉低利用率。

## 精度相关

### PC 侧 ORT（ONNX Runtime）

**是什么**：微软开源的 **ONNX 推理引擎**，在 PC 上常用 CPU 或 CUDA 跑同一份 ONNX，输出当 **参考基准**（Golden）。团队里常说「ORT 过了再谈上板」。

**打个比方**：用 **标准答案卷** 在电脑上先对一遍——确认图和权重在数学上没问题，再和板子比。

**在这句话里**：精度对比是 **板子上的 `.om` 输出** vs **PC 上 ORT 的输出**（或同一套测试集上的 mAP），看差多少。

---

### mAP（mean Average Precision，平均精度均值）

**是什么**：目标检测里常用的 **精度指标**。在固定测试集上，综合「框得准不准、查得全不全」算一个数（如 0.85）。**mAP 掉 8 个点** = 从比如 0.85 掉到 0.77 量级，业务上通常很严重，不是误差范围内抖一下。

**打个比方**：考试平均分从 85 掉到 77——不是某一道题粗心，是 **整体水平出了问题**。

**在这句话里**：说明板子上的结果和 PC 基准 **差很多**；团队先误以为是量化校准的问题。

---

### 校准集 / calib（Calibration Dataset）

**是什么**：做 **INT8 量化** 时用的一小批代表性图片（常见几十到几百张）。工具根据这批数据统计每层激活的范围，定 **scale**（缩放因子），把浮点映射到整数。**换 calib** = 换一批统计用的图，重新做 PTQ 量化。

**打个比方**：卖菜前先 **试称几筐** 定单价——换几筐试称，单价会变一点。

**在这句话里**：团队以为 mAP 掉点是 **量化标定没标好**，换了三版 calib 都没用——说明 **主因不在 calib**，而在编译日志里的算子路径（近似、fallback、融合失败）。若根本没做 INT8、只是 FP16，那换 calib 本来就不会有用，这里更多是 **排查方向错了**。

---

## 编译日志里的「元凶」

### Resize 走了近似 nearest

**是什么**：ONNX 里有个 **Resize** 算子，负责把特征图放大或缩小（改高宽）。`nearest` 是一种插值方式：新像素直接抄最近邻的老像素，不做平滑。**近似 nearest** 指芯片实现和 ONNX 标准 **不完全一致**——可能边界处理、rounding 不同，像素会差几个灰度值。

**打个比方**：把照片从 100×100 放大到 200×200，标准做法是「按规则抄格子」；近似版可能在边缘多抄或少抄一行， **肉眼看差不多，后面几十层卷积会放大差异**。

**在这句话里**：编译器选了 **快但不完全标准** 的实现；INT8 时这类层常常是 **精度掉点的嫌疑犯**。

---

### GridSample 标记了 CPU fallback

**是什么**：**GridSample** 是 ONNX 里的采样算子（按坐标从特征图上取点，常用于某些检测头、姿态、STN）。**CPU fallback** = 这颗算子 **NPU 不支持或没优化**，编译器把它安排到 **CPU 上算**，算完再把结果拷回 NPU。

**打个比方**：整条流水线本该在 **专用机床** 上加工，某一道工序只能拿到 **手工台** 上慢慢做，做完再搬回流水线——整体速度被这一站拖死。

**在这句话里**：GridSample 在 CPU 上跑 → **延迟高、MAC 利用率低**；若实现和 ORT 有差，还会 **贡献 mAP 掉点**。

---

### Transpose（转置）

**是什么**：改变张量 **维度的排列顺序**。最常见：图像从 **NCHW**（批次、通道、高、宽）变成 **NHWC** 或其他芯片喜欢的顺序。内存里同一块数， **读法变了**。

**打个比方**：同一柜书，原来按「作者→书名」排，Transpose 改成「出版社→作者」——内容没变，但 **找书方式全变**，要花时间重排。

**在这句话里**：编译器为了 Layout 自动 **插了四个 Transpose**——每次转置都要 **读一遍、写一遍 DDR**，又 **挡在 Conv 和 ReLU 中间**，后面说的融合链就断了。

---

### Conv+BN+ReLU 融合链

**是什么**：检测网络里极常见的一串层：**卷积（Conv）→ 批归一化（BN）→ ReLU 激活**。编译器理想情况是把它们 **合成一次硬件执行**：Conv 算完，BN 和 ReLU 在片上寄存器里顺手做完， **中间结果不写内存**。**融合链** 就是「本来可以连成一串、一次干完」的算子序列。

**打个比方**：炒菜本可以 **颠锅一次** 完成「下锅→调味→出锅」；融合链断了就变成 **炒完盛盘、再拿另一口锅加热、再盛盘**——多洗三个碗（多打 DDR）。

**在这句话里**：四个 Transpose 插在中间 → **Conv+BN+ReLU 合并不了** → 慢、费内存、利用率上不去。这是 **编译期图优化失败**，不是 runtime 能「临时合并」的。

---

## 两个容易混的概念

### 编译（compile）vs 推理 runtime

| | 编译（ATC 生成 `.om`） | 推理 runtime（ACL 等加载 `.om` 跑） |
|--|------------------------|-------------------------------------|
| **什么时候** | 开发机上，部署前，通常做一次 | 板子上，每次推理 |
| **干什么** | 选算子实现、融合、Tiling、Layout、写进 om | 加载 om、喂输入、取输出 |
| **能改什么** | 改 ONNX、改 ATC 参数、 **重编** | 改 batch、绑核、异步队列等 **很有限** |
| **打个比方** | 工厂 **排产 + 工艺定型** | 工人 **按工艺单开机生产** |

**在这句话里**：「不是 runtime 能修的」= Resize 近似、GridSample 上 CPU、融合被 Transpose 打断——这些在 **生成 om 时已写死**；你只能在板子上调 infer 参数， **改不了图里已经拆开的算子**。要修只能 **改 ONNX 或编译选项，重新 ATC**。

## 对照表：关键词 → 现象 → 该查哪

| 关键词 | 你看到的现象 | 第一反应该查什么 |
|--------|--------------|------------------|
| `.onnx` → `.om` | 以为部署完 | 还有 msprof、mAP、编译 log |
| msprof / MAC 25% | 慢 | log 里 fallback、Transpose、融合失败 |
| mAP 掉 8 点 | 不准 | 先 FP16 双端；再查近似 Resize、CPU 算子 |
| 换 calib 无效 | 量化不是主因 | 编译 log，别死磕校准集 |
| Resize approximate | 层输出和 ORT 有差 | 该层 dump 对比；考虑改 Resize 方式 |
| GridSample CPU fallback | 某层特别慢 | 改图替换算子或自研 NPU 算子 |
| Transpose ×4 | DDR 高、融合断 | 图 dump；导出前对齐 Layout |
| Conv+BN+ReLU 未融合 | 碎 kernel 多 | 挪走中间的 Transpose / 非常规 op |

## 读完后你应该能答的三句话

1. **`.onnx` 和 `.om` 差什么？**  
   ONNX 是通用计算图；om 是昇腾专用、编译期定好的执行方案，换芯片要重编。

2. **为什么 MAC 利用率低还会 mAP 掉？**  
   利用率低常因 fallback、融合断、Transpose 多；mAP 掉常因近似算子、CPU 路径和 ORT 不一致——两件事可能同时发生，根子都在 **编译 log**。

3. **为什么换 calib 没用？**  
   若问题来自 Resize 近似、GridSample fallback、融合失败，改校准集 **动的是量化 scale**，治不了 **算子路径错了** 这种病；要先 grep 编译 log，再决定改图还是重编。

## 术语速查（本篇专用）

| 术语 | 一句话 |
|------|--------|
| ONNX | 通用模型格式，「算什么」 |
| ATC | 昇腾编译器，ONNX → om |
| om | 昇腾离线可执行模型 |
| msprof | 昇腾性能分析工具 |
| MAC | NPU 里干卷积/矩阵乘的算力单元 |
| ORT | PC 上跑 ONNX 的引擎，常当基准 |
| mAP | 检测任务精度指标 |
| calib | INT8 量化用的统计图片集 |
| Resize | 改特征图尺寸的算子 |
| GridSample | 按坐标采样的算子 |
| CPU fallback | 算子改到 CPU 上算 |
| Transpose | 张量维度重排 |
| 融合 | 多层合并成一次执行，少写 DDR |
| runtime | 加载 om 做推理的程序，不改图结构 |
