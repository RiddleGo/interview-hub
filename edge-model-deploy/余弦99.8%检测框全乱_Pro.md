# 余弦 99.8%，检测框为什么还能全乱？

> **导读**：双端输出张量余弦 99.8%，画框却满天飞——别先怀疑量化；多数时候是后处理或某个 stride 分支出了问题，被全局平均数藏住了。  
> **阅读时间**：约 25 分钟  
> **适合谁**：学过一点深度学习、第一次做检测模型上板，或面试被问到「余弦很高但 mAP 崩了」的工程师。

---

## 排查现场：99.8% 能上线吗？——不能，我们比错节点了

某次 YOLOv5s 上昇腾 310，PC 侧 ORT FP16 当 Golden，和设备 INT8 `.om` 比 head 输出，整段张量余弦 99.82%。团队以为量化过关，开始压延迟。

C++ 后处理跑完一画框：小目标全漏，中等目标宽高颠倒，大目标偶尔还对。测试集 mAP 从 0.84 掉到 0.41。有人换 Entropy 校准、改 per-channel，精度不动。

把输出按 P3 / P4 / P5 三个 stride 分支拆开算余弦：P3 只有 0.93，P4、P5 都在 0.998 以上——全局 99.8% 是被大分支平均出来的。再查 C++ 解码：`wh` 误用了 sigmoid（应为 exp），letterbox 逆变换漏除 scale，框当然全乱。

这类翻车我见得不少：余弦是筛查工具，不是发货门禁；检测是局部敏感任务，全局指标很容易骗过你。

---

## 这个问题在部署链路里干什么

```
  .pt → ONNX → ORT 验收
              │
              ▼  PTQ + 校准集 → converter → .om
              │
              ▼  同 input.bin 上板推理
  ┌──────────────────────────────────────────────┐
  │ 张量对比：余弦 / MSE（本篇）                   │
  │ 误区：整段 head 一个余弦 → 99.x% 就过关       │
  └──────────────────────────────────────────────┘
              │
              ▼  decode → letterbox 逆变换 → NMS
  ┌──────────────────────────────────────────────┐
  │ 后处理（端侧 C++）                            │
  │ 框「全乱」多数在这里；张量余弦仍可很高         │
  └──────────────────────────────────────────────┘
              │
              ▼  mAP / 召回 / 坏例图
  业务门禁
```

张量余弦只回答：NPU 的 raw 输出和 PC Golden 像不像。框是后处理算出来的。正确顺序：输入一致 → 分分支比张量 → 比解码坐标 → 最后看 mAP。

## 零基础先搞懂：本篇会反复出现的名词

第一次读可扫一遍；后文用到时再回来查。名词按组排列，每组表后有一句「怎么用」。

### 1. 工具、文件与双端

| 名词 | 是什么 | 新人怎么理解 |
|------|--------|-------------|
| 双端 | PC + 开发板 | 比「同输入下输出一不一致」，不是比谁 mAP 高 |
| ORT | ONNX Runtime | PC 跑 ONNX 的引擎，常当 Golden |
| Golden | 基准输出 | PC 在标准输入下的推理结果，存 `output_gt.bin`；不是训练标签 |
| input.bin | 输入二进制 | 预处理后的张量裸落盘；无文件头，须约定 shape 和 float32 |
| `.om` | 昇腾芯片模型 | 由 ONNX 经 ATC 编译；类比 TensorRT 的 `.engine` |
| ADB | 调试桥 | `adb push/pull/shell`，PC 与板子传文件、跑推理 |
| CANN | 昇腾软件栈 | 编译器 + 算子库；日志里常有 fallback、Sigmoid 近似 |
| MD5 | 文件指纹 | input.bin 差一字节 MD5 就变；两端必须相同 |

表后：**精度对比的第一关是 input.bin 一致**。预处理只在 PC 做一次写 bin，设备侧 `fromfile` 读入，不要再 resize 或 `/255`。

**张量（tensor）**：多维数组，如 shape `(1, 3, 640, 640)` 是 1 张 RGB 图。**raw 输出**是网络最后一层直接吐出的数，还没 decode、还没 NMS——余弦比的是这里（或按分支比这里）。

### 2. 检测模型在预测什么

**目标检测**：输入一张图，输出多个矩形框，每个框带类别和置信度（conf，「有多像目标」）。

YOLO 类模型三段结构：

```
输入图 → Backbone（提特征）→ Neck（多尺度融合）→ Head（三尺度各预测一遍框）
```

**Head（检测头）**：在每个网格点上预测「有没有目标、框在哪、什么类」。量化、布局、解码问题最常出在 head。

**三分支与 stride**（640×640 输入、YOLOv5 系为例）：

| 分支 | stride | 特征图 | 约预测点数 | 主要负责 |
|------|--------|--------|-----------|----------|
| P3 | 8 | 80×80 | ~6400 | 小目标 |
| P4 | 16 | 40×40 | ~1600 | 中目标 |
| P5 | 32 | 20×20 | ~400 | 大目标 |

stride = 8 表示特征图走 1 格对应原图 8 像素；格子越密越适合小目标。P3 元素量约为 P5 的十几倍——整段展平算一个余弦时，P3 的错误会被 P4/P5「稀释」，这就是后文的平均效应。

**YOLOv5 每个预测点的 85 维是什么**（COCO 80 类为例，换数据集维数会变，结构相同）：

| 下标 | 含义 | decode 怎么用 |
|------|------|--------------|
| 0～1 | xy 偏移（raw） | sigmoid 后加 grid，再乘 stride → 框中心 |
| 2～3 | wh 偏移（raw） | exp 后乘 anchor 宽高 → 框宽、框高 |
| 4 | objectness | 「像不像有物体」；常与类分数相乘得 conf |
| 5～84 | 80 类分数 | argmax 或乘 objectness 得最终类别 |

**Anchor（锚框）**：每个网格点 preset 几组典型宽高；网络预测相对锚框的偏移。anchor 顺序或数值和 Python 差一处，框形状全怪，张量余弦仍可能很高。

### 3. 余弦、MSE、mAP：各回答什么问题

#### 余弦相似度

两个同 shape 张量拉平为向量 `a`、`b`：

```
cos(a, b) = dot(a, b) / (norm(a) × norm(b))
```

- `dot(a, b)`：对应元素相乘再求和（点积）  
- `norm(a)`：向量长度，`sqrt(sum(a[i] * a[i]))`  
- 代码里分母常加 `1e-12` 防除零（见附录 B）

余弦看「方向像不像」，对整体乘常数不敏感。部署里 0.99 表示大体像，**不能保证每个分支、每个前景格子都像**。适合逐层 / 逐分支筛查，不能单独当发货标准。

#### MSE（均方误差）

```
mse(a, b) = mean((a[i] - b[i]) * (a[i] - b[i]))
```

比余弦更抠每个元素的值。余弦高但 MSE 大，常是少数位置差很多（head 上高 conf 的点）。我会余弦 + MSE 一起看，必要时再加「高 conf 区 MSE」（附录 B）。

#### mAP 与业务指标

| 指标 | 含义 | 和余弦的关系 |
|------|------|-------------|
| mAP | 检测综合精度 | 发货门禁；余弦过了 mAP 仍可崩 |
| 召回率 | 该检出的检出了多少 | 漏检、阈值问题时会掉 |
| 误检率 | 不该有的框出了多少 | conf / NMS 阈值敏感 |

表后：**最终以 mAP 和坏例图为准**；余弦/MSE 用来定位问题在哪一层、哪一分支。

### 4. 后处理：raw 数怎么变成框

模型不直接输出像素坐标。decode 六步概览：

| 步 | 做什么 | 新人易错点 |
|----|--------|-----------|
| 1 | xy：sigmoid + grid + stride | C++ 手写 sigmoid 精度/溢出 |
| 2 | wh：exp × anchor | 误写成 sigmoid |
| 3 | conf：objectness × 类分数 | 阈值附近差 0.1 就漏检 |
| 4 | 按 conf_thres 过滤 | 与 Python 差 0.05，框数差很多 |
| 5 | letterbox 逆变换 | 漏除 scale、漏减 pad（第一大坑） |
| 6 | NMS 去重 | iou_thres、实现类型不一致 |

**letterbox 是什么**（新人必看）：

```
原图 1920×1080（宽>高）
    → 等比缩放到 640×360（塞进宽 640）
    → 上下各 pad 140 行灰边 → 640×640
    → 送进网络

推理后：框坐标在 640×640 的 letterbox 坐标系里
    → 必须减 pad、除 scale，才能映射回 1920×1080 原图
```

若端侧用 stretch 硬拉成 640×640（不等比），却按 letterbox 公式逆变换，框必偏。这和模型、量化无关。

**NMS（非极大值抑制）**：同一物体附近多个重叠框，按 conf 和 **IoU（交并比）** 留一个。IoU 直觉：

```
IoU = 两框交集面积 / 两框并集面积
```

`iou_thres` 和 Python 不一致时，会出现一端满屏框、一端全漏。**后处理 vs 预处理**：前者是出网络后（decode、NMS）；后者是进网络前（resize、归一化）。余弦只比后处理之前。

### 5. 量化与布局（排查靠后，但名词要先懂）

| 名词 | 含义 |
|------|------|
| FP32 / FP16 | 浮点；PC Golden 常用 |
| INT8 | 8 位整数；快、小，有离散误差 |
| PTQ | 训完再量化；校准集定 scale |
| skip quant | 指定层保持 FP16；head 最后 2～3 层常见 |
| scale（量化） | 浮点映到 INT8 的系数；**≠ letterbox 的 scale** |
| CPU fallback | 算子 NPU 不支持，改 CPU 算，可能有数值差 |
| NCHW | batch-通道-高-宽；PyTorch / 后处理常用 |
| NC1HWC0 | 昇腾原生布局；后处理前须转 NCHW，否则通道错位 |

```
                              ↓  ← 余弦 / MSE 比这里

                              ↓

                              ↓  ← 多数「框全乱」在这里

                              ↓

                              ↓  ← 业务门禁
```

## 为什么余弦很高，框却能全乱？

本质：余弦是全局平均，检测是局部敏感。

### 机制一：多分支平均

P4、P5 完全一致，仅 P3 余弦 0.50 时（P3 约占 head 元素七成）：

```
全局余弦 ≈ 0.70 × 0.50 + 0.27 × 1.0 + 0.03 × 1.0 ≈ 0.65
```

P3 余弦 0.98、P4/P5 为 1.0 时：

```
全局余弦 ≈ 0.70 × 0.98 + 0.27 × 1.0 + 0.03 × 1.0 ≈ 0.986
```

再略好一点就到 99.8%，但小目标分支已不可用。肉眼常见：小目标全漏，大目标还行——整体像「全乱了」。

### 机制二：背景淹没前景

head 上绝大多数格子是背景，conf 接近 0；决定检不检出的，是极少数前景高 conf 点。背景 99.9% 一致、前景 conf 差 0.1，全局余弦仍可能 99.5%+；过阈值 0.5 时却是检与不检的区别。所以要看高 conf 区 MSE，不能只看全局余弦。

### 机制三：后处理在张量之外

余弦比 raw tensor；框来自 decode + 逆变换 + NMS。下列问题张量余弦都可接近 100%：

| 问题 | 现象 |
|------|------|
| 类别 index 与训练 label 顺序不一致 | 框对、类错 |
| anchor 宽高写反 | 中心可能对、形状怪 |
| sigmoid 实现差 1e-3，卡在阈值边 | 框时有时无 |

第一反应：比的是哪段 tensor？后处理和 Python 是否逐步对齐？别先改 calib。

## 标准排查顺序

按「先易后难、先后处理后模型」走。

### 第一步：input.bin 一致

PC 预处理一次写 `input.bin`；设备只 `np.fromfile`，不再 resize。验证：

```powershell
# Windows 下看 MD5（Linux/macOS 用 md5sum）
certutil -hashfile input.bin MD5
```

两端 MD5 必须相同。不一致时，后面余弦无意义，还会误判成量化问题。

### 第二步：分 P3 / P4 / P5 比余弦 + MSE

禁止整段 head 展平算一个余弦。哪个分支明显偏低，就查该分支的布局、解码、量化。

```python
import numpy as np

def split_yolo_head(flat, shapes):
    """flat: float32 一维数组；shapes: 各分支 shape，须与 ONNX 一致"""
    parts, off = [], 0
    for sh in shapes:
        n = int(np.prod(sh))
        parts.append(flat[off : off + n].reshape(sh))
        off += n
    return parts

def cosine_mse(a, b):
    af = a.flatten().astype(np.float64)
    bf = b.flatten().astype(np.float64)
    cos = np.dot(af, bf) / (np.linalg.norm(af) * np.linalg.norm(bf) + 1e-12)
    mse = np.mean((a.astype(np.float64) - b.astype(np.float64)) ** 2)
    return float(cos), float(mse)
```

若 P3 < 0.96 而 P4/P5 > 0.99，全局 99.8% 就有解释，范围缩到 P3。

### 第三步：后处理对齐（多数问题在这里）

**3.1 解码逐步对照**

| 步骤 | 在算什么 | 常见不一致 | 现象 |
|------|----------|-----------|------|
| xy | sigmoid(raw) + grid，× stride | sigmoid 精度 | 框整体偏移 |
| wh | exp(raw) × anchor | 误用 sigmoid | 宽高全怪 |
| grid | 网格 (i,j) 起算方式 | 差半格 | 固定像素偏移 |
| 类别 | class argmax | names 顺序错 | 位置对、类错 |

**怎么查**：固定一张图、一个网格 (i,j)、一个 anchor index；Python 与 C++ 打印同位置的 raw 和解码后 `(x,y,w,h)`。数值对不上 = 后处理，与模型无关。

**3.2 letterbox 逆变换**

预处理记录：

```
scale = min(640 / 原图高, 640 / 原图宽)
pad_w, pad_h = 左右或上下补边的像素数
```

解码后在 letterbox 640×640 坐标系得到 `(x, y, w, h)`，映射回原图：

```
x_orig = (x - pad_w) / scale
y_orig = (y - pad_h) / scale
w_orig = w / scale
h_orig = h / scale
```

常见错误：stretch resize 却用 letterbox 逆变换；只除 scale 不减 pad；宽高忘除 scale。验证时比 NMS 前坐标数值，不要只肉眼看图。

**3.3 阈值与 NMS**

核对 `conf_thres`、`iou_thres`、NMS 类型（普通 / Soft / DIoU）与 Python 验证脚本一致。

### 第四步：head 量化与布局

head 最后几层对 INT8 敏感，conf 在 0.4～0.6 附近误差会被放大。常见 head 最后 2～3 层 skip quant 保持 FP16；校准集要有足够正样本。

部分 CANN 版 Sigmoid 快速近似在 0.4～0.6 误差大——恰是 conf 阈值区。可换标准实现或 skip 该层对比。

NC1HWC0 → NCHW 转错时，余弦有时仍高，通道却错位。用 Golden 与转换结果 **同一 (n,c,h,w) 下标** 逐项 diff。

### 第五步：算子与量化深层

前两步都干净再查：Resize 近似、校准集分布偏、CPU fallback。逐层余弦从哪层掉到 98% 以下，从该层往前查 converter。这通常是少数 case。

## 四层验证门禁

| 层级 | 做什么 | 通过标准（示例，视项目调整） |
|------|--------|------------------------------|
| L1 输入 | 同 input.bin，MD5 一致 | 字节级相同 |
| L2 张量 | 分 P3/P4/P5 余弦 + MSE | 各分支 cos > 0.99 |
| L3 解码 | 关 NMS，比 decode 后坐标 | top-K 框 IoU 或坐标差达标 |
| L4 业务 | 全量后处理 + 测试集 | mAP / 召回 / 误检在业务线内 |

铁律：余弦过了，业务不一定过；业务过了，关键层余弦一般过得去。

## 最常见坑

**坑 1：整段 head 一个余弦**

做法：`(1, 25200, 85)` 展平算 cos=0.998。现象：mAP 崩。原因：P3 只有 0.92 被拉高。改：三分支拆开，修低分支。

**坑 2：letterbox 逆变换漏了**

做法：当 640 就是原图。现象：框整体偏或缩放错。原因：没除 scale、没减 pad。改：对齐 Python `scale_coords`。

**坑 3：wh 用 sigmoid**

做法：xy 和 wh 同一套激活。现象：框又扁又宽。原因：wh 应为 `exp(raw) * anchor`。改：一行激活。

**坑 4：类别顺序和 names 不一致**

做法：端侧按字母序排类。现象：框对、类错，mAP 接近 0。改：沿用训练 yaml 的 class 顺序。

**坑 5：NC1HWC0 转 NCHW 错位**

做法：memcpy 步长错。现象：余弦 0.97～0.99，框天马行空。查：固定 (h,w) 看各 channel 与 Golden 是否一致。

**坑 6：head 全 INT8 + Sigmoid 近似**

做法：为体积全 INT8。现象：余弦 99%+，conf 在 0.5 附近系统性偏。改：head 最后几层 FP16。

## 面试追问

### 1

**问：** 余弦相似度 99.8%，检测框全乱，你第一反应查什么？

**答：** 先确认比的是整段 head 还是分 stride 分支；常是某分支坏了被平均掉。再查 C++ 后处理与 Python：decode 的 sigmoid/exp、letterbox 逆变换、类别顺序和 conf/NMS 阈值。这些对了再查 head 量化和 NC1HWC0，别先调校准集。

---

### 2

**问：** 为什么 YOLO 要多 stride 分开算余弦，不能整段展平？

**答：** P3/P4/P5 元素量差一个数量级，P3 占大头。一分支错，全局余弦仍可能 99%+，小目标或中目标已全挂。分开算才能定位 P3、P4 还是 P5。

---

### 3

**问：** letterbox 逆变换写错，张量余弦会受影响吗？

**答：** 不会。余弦比 raw 输出；逆变换在后处理里。会出现余弦 99.8%、框全偏——说明网络像 Golden，问题在坐标映射。应比 decode 后、NMS 前坐标。

---

### 4

**问：** 检测 head 为什么要 skip quant？

**答：** head 输出范围小、对 conf 阈值敏感，INT8 误差在 0.4～0.6 会被放大；叠 Sigmoid 近似时全局余弦仍可能高。常见 head 最后 2～3 层 FP16，用一点体积换 mAP 稳定。

---

### 5

**问：** 验收时张量指标和业务指标，哪个说了算？

**答：** 业务指标。mAP、召回、误检、坏例图是发货门禁；余弦/MSE 是排查工具。余弦过了业务可以不过；业务过了，关键层余弦一般也过得去。

## 附录 B：分分支余弦 + 高 conf 区域 MSE

**前置**：同 `input.bin` 在 PC 得 `output_gt.bin`，在设备得 `output_npu.bin`。  
**BRANCH_SHAPES** 须与 Netron 里 output 节点一致。

```python
import numpy as np

# ---------- 按项目修改 ----------
BRANCH_SHAPES = [
    (1, 3, 80, 80, 85),   # P3: batch, 每格3 anchor, 80×80, 85维/anchor
    (1, 3, 40, 40, 85),   # P4: stride 16
    (1, 3, 20, 20, 85),   # P5: stride 32
]
CONF_INDEX = 4           # 85 维里 objectness 下标（YOLOv5 常见为 4）
CONF_THRESH = 0.01       # 高于此视为「可能是前景」


def load(path, shapes):
    flat = np.fromfile(path, dtype=np.float32)
    parts, off = [], 0
    for sh in shapes:
        n = int(np.prod(sh))
        parts.append(flat[off : off + n].reshape(sh))
        off += n
    if off != flat.size:
        raise ValueError(f"shape 与文件不符: 文件 {flat.size} 元素, 配置 {off}")
    return parts


def cosine_mse(a, b):
    # cos(a,b) = dot(a,b) / (norm(a)*norm(b))
    af, bf = a.flatten().astype(np.float64), b.flatten().astype(np.float64)
    cos = np.dot(af, bf) / (np.linalg.norm(af) * np.linalg.norm(bf) + 1e-12)
    mse = np.mean((a.astype(np.float64) - b.astype(np.float64)) ** 2)
    return float(cos), float(mse)


def high_conf_mse(a, b, conf_idx=CONF_INDEX, thresh=CONF_THRESH):
    """只在任一侧 conf > thresh 的格子上算 MSE；若远大于全局 MSE，说明背景把指标洗好看了"""
    ca, cb = a[..., conf_idx], b[..., conf_idx]
    mask = (ca > thresh) | (cb > thresh)
    if not mask.any():
        return float("nan")
    return float(np.mean((a[mask] - b[mask]) ** 2))


if __name__ == "__main__":
    gt_parts = load("output_gt.bin", BRANCH_SHAPES)
    npu_parts = load("output_npu.bin", BRANCH_SHAPES)
    names = ["P3/stride8", "P4/stride16", "P5/stride32"]

    print("=== 分分支对比（验收看各分支，不看整段） ===")
    for name, g, n in zip(names, gt_parts, npu_parts):
        cos, mse = cosine_mse(g, n)
        hmse = high_conf_mse(g, n)
        print(f"{name}: cos={cos:.6f}  全局mse={mse:.6e}  高conf区mse={hmse:.6e}")

    all_gt = np.concatenate([p.ravel() for p in gt_parts])
    all_npu = np.concatenate([p.ravel() for p in npu_parts])
    cos_all, _ = cosine_mse(all_gt, all_npu)
    print(f"\n整段展平 cos={cos_all:.6f}  ← 仅参考，不能代替分分支验收")
```

## 术语速查

| 术语 | 含义 | 直觉 |
|------|------|------|
| 张量 | 多维数组 | shape 如 (1,3,640,640) |
| 余弦 | 向量夹角相似度 | 全局平均，易掩盖坏分支 |
| MSE | 均方误差 | 抠每个元素差多少 |
| stride | 下采样步长 | 8/16/32 → 小/中/大目标 |
| P3/P4/P5 | 三个检测分支 | 80×80 / 40×40 / 20×20 |
| head | 检测头 | 出框预测；量化最敏感 |
| anchor | 锚框 | 预设宽高模板 |
| decode | 解码 | raw → (x,y,w,h,conf,class) |
| letterbox | 等比缩放+补边 | 推理后须逆变换回原图 |
| conf | 置信度 | 阈值附近差 0.1 可漏检 |
| NMS | 非极大值抑制 | 重叠框去重 |
| IoU | 交并比 | 两框重叠程度 |
| mAP | 检测综合精度 | 发货门禁 |
| Golden | PC 基准输出 | output_gt.bin |
| input.bin | 预处理输入落盘 | 两端 MD5 须一致 |
| INT8 / FP16 | 整数量化 / 半精度浮点 | head 常保留 FP16 |
| PTQ | 训后量化 | 校准集定 scale |
| NC1HWC0 | 昇腾布局 | 须转 NCHW 再后处理 |
| 后处理 | decode + 逆变换 + NMS | 余弦比它之前 |
