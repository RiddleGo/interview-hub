# 从训练到边缘上线，你走什么链路？

> **导读**：训练完 `.pt` 不等于能发货——中间还有导出、改图、量化、编译、上板、业务验收七道关；每关有交付物和回滚点，出了问题才知道该找算法还是部署。  
> **阅读时间**：约 25 分钟  
> **适合谁**：学过 PyTorch、还没系统走过「训完 → 芯片模型 → 设备推理」全链路，或面试被问到「部署流程你怎么讲」的工程师。

---

## 交付物界定：口头说「上线了」，文件夹里其实有三种模型

某次检测项目联调，算法说「模型训好了」，部署说「量化也过了」，测试说「板子上框全是乱的」。三方各掏一份文件：`.pt` 权重、INT8 ONNX、`.om` 离线模型——三个版本、三套预处理脚本、CANN 还是两个月前的。

新人对着余弦 0.97 的日志发呆：张量看着像，mAP 掉了八个点。量化工程师改了一周校准集，驱动同事查了一周 layout，最后发现是 **C++ 后处理 letterbox 逆变换和 Python 训练脚本差了一个像素**。更糟的是，没人能说清问题该落在链路的哪一段——因为项目从头到尾没按阶段验收，也没有明确的回滚点。

我习惯把「从训练完到边缘上线」拆成 **7 个阶段**，每段有 **可交付物、验收标准和回滚点**。岗位不同负责段不同，但 **必须能判断问题落在哪一段**——否则量化工程师和驱动工程师会互相甩锅一周。下面按从前到后的顺序讲，前一阶段验收不过，绝不进下一阶段。

---

## 全链路在干什么

```
  阶段1          阶段2              阶段3                 阶段4
  .pt ──► FP32/FP16 ONNX ──► 芯片适配 ONNX ──► INT8 ONNX + PC 预验 ──► .om 离线模型
   │         导出+checker        算子兼容+改图         PTQ+ORT对比              ATC编译
   │                                                                               │
   │         回滚：.pt权重    回滚：原始ONNX      回滚：FP16适配版        回滚：量化ONNX
   │                                                                               │
   └───────────────────────────────────────────────────────────────────────────────┘
                                          │
                    阶段5                    阶段6                    阶段7
              单板双端张量对齐  ──►  业务指标+坏例回归  ──►  版本锁定+交付归档
              同 input.bin 对比         mAP/NMS/压测           torch/CANN/驱动锁死
                    │                        │                        │
              回滚：上版合格.om        回滚：上版达标模型        回滚：上一正式版
```

整条链的核心价值，不只是「把模型跑起来」，而是 **可控、可复现、可排查**。每个阶段把大问题拆成小问题；出了故障按阶段顺推，不用全链路盲目 grep。

## 零基础先搞懂：本篇会反复出现的名词

### `.pt`、ONNX、芯片模型，各是什么

**`.pt`（PyTorch checkpoint）** 绑训练框架：算子实现、版本、自定义 op 都在 PyTorch runtime 里。车机、相机 SoC 上往往装不了，或装了也占内存。

**ONNX（Open Neural Network Exchange）** 是中间合同：图结构 + 权重 + 输入输出名。PC 上用 **ORT（ONNX Runtime）** 做 Golden 验收，也给厂商 converter 吃。ORT 能跑，只说明图在数学上成立，不说明 NPU 吃得下。

**芯片模型**（昇腾 `.om`、TensorRT `.engine`、SNPE `.dlc` 等）是 **已经 lowering 过** 的二进制：算子映射到 NPU 指令、layout 可能从 NCHW 改成 NHWC、该 fuse 的 fuse 了、该 tile 的 tile 了。团队里 ORT FP32/FP16 是 **验收基准**，不是交付物；交付物是芯片侧二进制 + 锁版本的工具链。

### PTQ 与 PC 侧预验

**PTQ（Post-Training Quantization，后训练量化）** 在训练完成后，用校准集统计激活范围，把 FP32/FP16 权重和激活映射到 INT8。多数项目先 PTQ 上车；掉点超业务线再考虑 QAT 或混合精度。

**PC 侧预验** 指在不上板之前，用 ORT 直接跑 INT8 ONNX，对比 FP16 的精度和业务指标。能筛掉大部分量化问题，省联调时间——但 PC 过了不代表板端一定过，converter 和驱动还会引入差异。

### 可交付物与回滚点

**可交付物** 是阶段结束时必须归档的文件和报告，下阶段只认这些输入。**回滚点** 是验收失败时退回的稳定版本——不是「重新训一遍」，而是回到上一阶段已验收通过的产物。没有回滚点，联调就会变成「不知道哪个版本是对的」。

### 双端对比与 Golden

**双端对比** 指 PC 和设备在 **同一份 `input.bin`** 下各跑推理，比输出张量的余弦相似度和 MSE。**Golden** 是 PC 侧在标准输入下的参考输出。精度验证的第一原则是 **控制变量**：唯一允许不同的是推理设备，输入、预处理、后处理参数都应一致。

## 阶段 1：训练模型导出标准化 ONNX

### 干什么

从训练框架的 **动态训练图**，转成固定、规范的通用推理中间表示。这是所有后续工作的基础——导出错了，后面改图、量化、编译全是在歪图上补洞。

具体要做的事：切换 `model.eval()`，关掉 Dropout、BatchNorm 训练态；固定 **静态输入 shape**（边缘部署优先杜绝动态 shape，编译器才能做满内存规划，延迟抖动也小）；统一 ONNX Opset 版本（昇腾生态常见 11/13，但以你当前 ORT + converter 组合实测为准）；规范算子命名，和训练代码对应，方便后面定位；在训练框架内 **手动折叠 BN 层**，减少编译期融合失败；最后跑 `onnx.checker` 语法校验。

### 怎么验收

- 导出 ONNX 能在 ORT 里跑通，输出和 PyTorch 同输入下数值接近
- Netron 打开无 `aten::` 残留、无异常 Constant 节点
- 文档记录：训练框架版本、Opset、导出脚本、静态 shape

### 典型坑

**做法**：导出时开着 `dynamic_axes`，或训练脚本里还有 `torch.onnx.export` 没关的随机增强。  
**现象**：converter 报 dynamic shape 警告，或每换一次分辨率就慢路径。  
**原因**：NPU 对动态轴支持往往不完整，能编过不代表跑得漂亮。  
**怎么查**：Netron 看 input 维度有没有 `-1`；converter 日志搜 `dynamic`。  
**怎么改**：固定业务分辨率，必要时多尺寸就导出多份静态模型。

**做法**：自定义算子没注册 ONNX 符号。  
**现象**：`checker` 通过但 converter 直接 unsupported。  
**原因**：PyTorch 导出了 `aten::` 或 opaque 自定义 op。  
**怎么查**：`grep` ONNX 里有没有非标准算子名。  
**怎么改**：换成标准 op 组合，或写自定义 ONNX op 并在 converter 侧注册。

### 交付与回滚

- **交付物**：干净的 FP32/FP16 ONNX、导出环境说明
- **回滚点**：训练框架原始权重 `.pt`
- **责任**：算法/训练工程师；典型问题是动态算子、自定义 op、shape 不固定、Opset 不兼容

## 阶段 2：算子兼容性校验与图结构适配

### 干什么

把通用 ONNX 改造成目标芯片 **「看得懂、跑得好」** 的结构。是从「能编译」到「性能好」的关键前置——官方文档写「支持 Softmax」，不会写「第 2 维做 Softmax 整层 6ms」。

对照 **算子知识库** 和厂商官方清单，做全量支持性检查；对不兼容算子做等价替换（大核卷积拆小核堆叠、中间维 Softmax 转置到末位、Resize 插值方式对齐等）；主动做图结构优化——把 Transpose、Cast、Reshape 等 **融合阻断算子** 移到算子链首尾，引导编译器最大化 fuse；对齐硬件约束，比如通道数 pad 到 16 的整数倍，消除非对齐带来的算力浪费。

### 怎么验收

- converter 不再报 unsupported（或已知 unsupported 已用替代方案并记录）
- ORT 跑适配后 ONNX，和阶段 1 输出对比，确认改图没改数学语义
- 存档：替换了哪些算子、为什么、精度影响评估

### 典型坑

**做法**：Netron 看到 unsupported 就直接改模型结构，没先在 PC 验证。  
**现象**：编译过了，双端精度崩。  
**原因**：等价替换在浮点下就不完全等价，或 INT8 下更敏感。  
**怎么查**：改图前后 ORT 对比同一输入的输出余弦。  
**怎么改**：每次 graph surgery 后 ORT 回归，再进 converter。

**做法**：为了「消除 Transpose」强行改 layout，和训练权重对不上。  
**现象**：权重加载失败或输出全乱。  
**原因**：改了输入 layout 但没同步改权重排列。  
**怎么查**：对比改图前后 Conv 权重的 shape 和数值分布。  
**怎么改**：按厂商文档的 layout 约定改，或让 converter 自动转。

### 交付与回滚

- **交付物**：芯片适配后的 ONNX、算子适配说明
- **回滚点**：阶段 1 的原始导出 ONNX
- **责任**：部署工程师；典型问题是算子不支持、结构不符合硬件约束、融合条件不满足

## 阶段 3：PTQ 离线量化与 PC 侧精度预验证

### 干什么

在 PC 侧完成量化和初步精度验证，不用上板就能筛掉大部分量化问题。

制备 **校准集（calibration set）**：覆盖业务典型场景，多数项目在 100～500 张，分类可偏少、检测分割建议更多，分布要和线上真实场景一致——校准集不是训练集，也不是随便抓的图库。执行 PTQ，生成 INT8 ONNX；对量化敏感层配置 `skip_layers` 做混合精度。用 ORT 跑 INT8，对比 FP16 的精度和业务指标；逐层对比输出，迭代 skip 列表。 **PC 侧 INT8 精度在业务可接受范围内，再进板端**——PC 就崩了，上板只会更惨。

### 怎么验收

- INT8 vs FP16 的关键层余弦、任务指标掉点在业务线内
- 量化配置表完整：算法、校准集说明、skip 列表、校准集 hash
- 有 PC 侧预验证报告，不是口头「感觉还行」

### 典型坑

**做法**：校准集全是实验室白底图，线上是复杂场景。  
**现象**：PC 指标好看，上板或坏例全崩。  
**原因**：激活统计分布和真实场景偏离，scale 定偏了。  
**怎么查**：对比校准集和线上数据的亮度/场景分布；看敏感层 histogram。  
**怎么改**：补 hard case、过曝/欠曝、小目标到校准集；必要时 QAT 或加 skip。

**做法**：PC 侧 INT8 不过，直接上板调。  
**现象**：板端联调一周，问题复现不了。  
**原因**：板端还有 converter、layout、驱动差异，和量化搅在一起。  
**怎么查**：先在 ORT 把 INT8 调到可接受，再进阶段 4。  
**怎么改**：严格阶段门禁，别跳步。

### 交付与回滚

- **交付物**：量化 ONNX、量化配置表、PC 侧预验证报告
- **回滚点**：阶段 2 的 FP16 适配版 ONNX
- **责任**：量化/部署工程师；典型问题是掉点严重、校准集分布不符、敏感层定位

---

## 阶段 4：芯片离线编译与性能深度优化

### 干什么

把 ONNX 翻译成芯片可直接执行的离线指令文件，集中完成编译期优化——这是 **性能优化的核心阶段**。

用厂商工具（昇腾 **ATC**、TensorRT `trtexec` 等）编译为芯片专属模型；开全量优化：最高优化级别、融合开关、AutoTune 搜最优 Tiling 分块。**完整留存编译日志**，重点归档 CPU fallback、算子近似实现、融合失败等 Warning——精度/性能扯皮时，日志是第一依据。做性能初测：单算子、整网延迟，看 MAC 利用率和 DDR 带宽是否在合理区间；不达标就迭代融合策略、Tiling、高代价算子替换。

### 怎么验收

- 编译成功，无未处理的 fatal warning
- 性能初测报告：延迟、MAC 利用率、带宽（区分 batch=1 延迟和 batch=N 吞吐）
- 高危 warning 已归档并有人读过

### 典型坑

**做法**：编译通过就以为 INT8 真在 MAC 上跑。  
**现象**：INT8 比 PC FP32 还慢。  
**原因**：假量化——图里名字是 INT8，大量 Q/DQ 或 Cast 在 FP16/FP32 和 INT8 之间倒，或关键 op CPU fallback。  
**怎么查**：convert 日志看量化层数；profiler 看 Precision 列和 Device 列。  
**怎么改**：改图减少 Cast、换支持 INT8 的 op 路径、重 tune tiling。

**做法**：Warning 全当噪音忽略。  
**现象**：双端余弦 98%，框却飘——日志里某 Resize 用了近似实现。  
**原因**：converter  silently 近似了算子。  
**怎么查**：编译日志搜 `approximate`、`fallback`、`warning`。  
**怎么改**：换等价 op 或接受近似并在阶段 5 逐层验证。

### 交付与回滚

- **交付物**：可执行 `.om`（或等价芯片模型）、编译日志+高危 warning 归档、性能初测报告
- **回滚点**：阶段 3 的量化 ONNX
- **责任**：部署工程师；典型问题是编译失败、性能远低于预期、CPU fallback

## 阶段 5：单板推理与双端张量精度对齐

### 干什么

在真实硬件上验证推理正确性。核心原则是 **单一变量控制**——只有推理设备不同，其余完全一致。

PC 侧生成标准 `input.bin`：所有预处理（resize、归一化、通道转换） **只在 PC 做一次**，端侧仅读 bin 推理。ADB（或厂商等价工具）推送模型、输入、推理程序；单板执行，拉回 `output.bin`。先校验输入 MD5 一致，再比输出余弦和 MSE。不达标就层输出 Dump，逐层对比，区分算子实现、量化、layout 哪类问题。

建议顺序：先 **FP16/FP32 双端对齐**（问题在 converter，和 INT8 无关），再比 INT8；别 FP16 不过就调校准集。

### 怎么验收

- 同 `input.bin` 下，关键层余弦达到团队约定门槛（检测 head 常比 backbone 更严）
- 有张量对比报告；有问题时有逐层排查记录

### 典型坑

**做法**：两端各读 jpg 各做预处理。  
**现象**：余弦偏低，校准集怎么换都没用。  
**原因**：OpenCV 和 PIL 的 resize 差 0.1 像素，INT8 会放大。  
**怎么查**：两端进网络的 tensor 做 md5。  
**怎么改**：PC 生成 bin，设备只 `fromfile`。

**做法**：全局余弦 99%+，就认为能发货。  
**现象**：框全乱。  
**原因**：YOLO 多 stride，坏一个分支被 backbone 大 tensor 平均掉。  
**怎么查**：按 stride/分支拆开算余弦。  
**怎么改**：head 单独对比；网络对齐后再查后处理。

### 交付与回滚

- **交付物**：单板张量精度对比报告、逐层排查记录（如有）
- **回滚点**：上一版验证合格的 `.om`
- **责任**：部署工程师；典型问题是张量一致性差、layout 错误、驱动实现差异

## 阶段 6：业务指标验收与坏例全量回归

### 干什么

从「张量对」到「业务效果对」——也是最容易出现 **「余弦很高但框全乱」** 的阶段。

跑完整端到端 Pipeline，全量测试集统计 mAP、准确率、召回、误检率等。**对齐后处理逻辑**：坐标解码、anchor 参数、letterbox 逆变换、NMS 算法和训练端完全一致——张量对齐了，后处理差一个像素，框也能全飘。Bad Case 逐一排查：是模型本身、量化引入，还是部署引入。再做性能与稳定性压测：连续推理延迟、吞吐、功耗、长时间运行，确认满足产品规格。

判断口诀：**张量对齐但业务掉点 → 先查后处理，再联合算法；张量不对导致业务掉点 → 部署侧问题**。

### 怎么验收

- 业务指标相对 FP32 掉点在约定范围内
- 坏例有分类结论（模型/量化/后处理/数据）
- 性能功耗报告满足产品规格

### 典型坑

**做法**：C++ 后处理从 Python 「翻译」时改了 NMS 阈值或 sigmoid 实现。  
**现象**：最后一层余弦 99.8%，mAP 掉很多。  
**原因**：后处理不在网络图里，双端对比比不到。  
**怎么查**：用同一网络输出，Python 和 C++ 各跑一遍后处理比框。  
**怎么改**：后处理参数表和训练脚本 lock 一致，写进 release note。

### 交付与回滚

- **交付物**：业务验收报告、性能功耗报告、Bad Case 分析
- **回滚点**：上一版业务达标的模型版本
- **责任**：算法 + 部署联合

---

## 阶段 7：版本锁定与上线归档

### 干什么

锁死全链路变量，保证 **可复现、可回滚**——项目交付最后一道关。

形成唯一交付包：原始 ONNX、量化 ONNX、离线 `.om`；工具链版本（PyTorch、Opset、CANN/TensorRT、驱动固件）；编译参数、量化配置、skip 列表；标准测试输入、基准输出、完整验收报告。更新团队算子知识库，补充本次踩坑和最优 Tiling/融合参数。上线后保留回滚路径，异常时快速切回上一稳定版。

半年后要能复现当时的余弦和 mAP——靠的是这份包，不是某同事笔记本上的临时目录。

### 怎么验收

- 交付包清单完整，新人能按文档复现验收结果
- 版本说明文档已评审
- 回滚演练做过或至少有明确步骤

### 典型坑

**做法**：只归档 `.om`，没锁 CANN 和驱动。  
**现象**：同版本复现不出之前的测试结果。  
**原因**：驱动小版本改了算子实现。  
**怎么查**：对比两次环境的 CANN、固件、编译参数 hash。  
**怎么改**：交付包写全版本号；CI 里固定工具链镜像。

### 交付与回滚

- **交付物**：完整上线交付包、正式版本说明
- **回滚点**：上一个正式上线的稳定版本
- **责任**：部署工程师牵头，全团队确认验收

## 问题落在哪一段？快速定位表

| 现象 | 归属阶段 | 核心负责岗位 |
|------|----------|--------------|
| ONNX 导出报错、checker 不通过 | 阶段 1 | 算法工程师 |
| 编译算子不支持、参数超限 | 阶段 2 | 部署工程师 |
| PC 侧 ORT 跑 INT8 精度就大幅下降 | 阶段 3 | 量化/部署工程师 |
| 编译通过但整网性能远低于预期 | 阶段 4 | 部署工程师 |
| 同 input 下双端张量余弦明显偏低 | 阶段 5 | 部署工程师 |
| 张量对齐但业务指标掉点 | 阶段 6 | 先查后处理，再联合算法 |
| 同版本复现不出之前的测试结果 | 阶段 7 | 版本未锁定，查工具链/驱动 |

表后补一句：这张表是 **协作边界**，不是甩锅清单。你定位到阶段 5，仍可能需要算法帮看某层是否该 skip；关键是 **别在阶段 3 无限调校准集，而实际是阶段 2 改图改错了**。

## 最常见坑（跨阶段）

### 坑 1：跳过 PC 侧 INT8 预验，直接上板

典型做法：量化工具出 INT8 ONNX 就扔给 converter。现象：板端和 PC 结论打架，不知道怪量化还是怪驱动。原因：阶段 3、4、5 的问题叠在一起。怎么查：ORT 能否复现 INT8 掉点；converter 日志有无 fallback。怎么改：阶段门禁，PC INT8 过关再编译上板。

### 坑 2：没有分 FP16 和 INT8 两道门禁

典型做法：一上来就比 INT8 余弦。现象：FP16 双端就差一截，却去改校准集。原因：converter/layout 问题和量化无关。怎么查：同 bin 先比 FP16 `.om` 和 ORT FP16。怎么改：FP16 对齐 → INT8 对齐 → 业务指标，顺序不能乱。

### 坑 3：各阶段产物版本混用

典型做法：阶段 2 改了图，阶段 3 还用旧 ONNX 做量化。现象：编译的模型和报告对不上。原因：没把交付物当唯一输入。怎么查：文件 hash 和阶段记录对一下。怎么改：每阶段输出进版本目录，命名带阶段号。

### 坑 4：验收只有 mAP，没有张量对比

典型做法：只看业务指标。现象：mAP 掉了不知道怪量化还是后处理。原因：业务指标离网络输出隔了好几层。怎么查：先比最后一层网络输出，再比加后处理后的框。怎么改：分层验收，张量门禁在前，业务门禁在后。

建议排查总顺序：**导出 checker → ORT 浮点对齐 → 改图后 ORT 回归 → PC INT8 预验 → 编译日志 → FP16 双端 → INT8 双端 → 后处理与 mAP → 版本归档**。

## 面试追问

### 1

**问：** 从训练完到边缘上线，你怎么口述整条链路？

**答：** 我按七段说，每段有交付物和回滚点：导出 ONNX（静态 shape、命名、checker）→ 算子/结构兼容改图 → PTQ + 校准 → ORT 上先验 INT8 → 转芯片模型 → 设备推理 → 同 input 双端对比 → 业务指标 + 坏例回归 → 锁版本（torch/onnx/opset/toolchain/driver）。岗位不同负责段不同，但必须能判断问题落在哪一段，否则量化和对驱动会互相甩锅。

### 2

**问：** 为什么 PC 侧 INT8 过了，板端还可能不过？

**答：** PC 上 ORT 和 NPU 不是同一个 runtime。converter 可能插 Cast、改 layout、近似算子，驱动还有实现差异。所以阶段 3 是筛量化问题，阶段 5 还要同 bin 做双端对比。顺序上先 FP16 双端再过 INT8，别把 converter 问题和校准集搅在一起。

### 3

**问：** 张量余弦 99% 能上线吗？

**答：** 不能单凭这一个数。余弦对缩放不敏感，全局 99% 可能掩盖检测 head 某条分支已经坏了。我会拆 head/backbone 看余弦和 MSE，再上 mAP 和坏例图。张量对齐是必要条件，发货还要看业务指标、延迟和功耗。

### 4

**问：** 张量完全对齐，业务指标却掉了，你先查什么？

**答：** 先查后处理，不是先加校准图。对 C++ 里 letterbox 逆变换、NMS 阈值、sigmoid/exp 解码是否和 Python 训练一致；用同一网络输出分别跑 Python 和 C++ 后处理比框。网络最后一层对齐后还掉点，再联合算法看模型或坏例数据。

### 5

**问：** 阶段 2 改图和阶段 3 量化，怎么划分责任？

**答：** 阶段 2 解决「芯片吃不吃得下、结构合不合约束」，交付适配 ONNX，改图前后 ORT 要回归；阶段 3 解决「INT8 scale 定得准不准」，交付量化 ONNX 和 PC 预验报告。若 PC 上 FP16 改图后就偏了，别进量化；若 FP16 对齐 INT8 才掉点，在阶段 3 调校准集和 skip_layers。

## 附录 B：阶段 1 最小导出与校验脚本

复制即可理解导出阶段在做什么；按项目改 `MODEL_PATH`、`INPUT_SHAPE` 和 `forward` 调用。

```python
import torch
import onnx
import onnxruntime as ort
import numpy as np

# ---------- 配置 ----------
MODEL_PATH = "best.pt"          # 或你的 nn.Module
ONNX_OUT = "model_fp16.onnx"
INPUT_SHAPE = (1, 3, 640, 640)  # NCHW 静态
OPSET = 13                      # 以 ORT + converter 实测为准
INPUT_NAME = "images"
OUTPUT_NAMES = ["output"]


def export_and_check():
    # 示例：若已是 nn.Module 则直接 load；.pt 按项目写法调整
    model = torch.load(MODEL_PATH, map_location="cpu")
    if isinstance(model, dict):
        raise ValueError("请改为加载 state_dict 或完整 Module 的写法")
    model.eval()

    dummy = torch.randn(*INPUT_SHAPE)
    torch.onnx.export(
        model,
        dummy,
        ONNX_OUT,
        opset_version=OPSET,
        input_names=[INPUT_NAME],
        output_names=OUTPUT_NAMES,
        dynamic_axes=None,  # 边缘部署优先静态
    )
    onnx.checker.check_model(ONNX_OUT)
    print(f"checker 通过: {ONNX_OUT}")

    # ORT 试跑
    sess = ort.InferenceSession(ONNX_OUT, providers=["CPUExecutionProvider"])
    x = dummy.numpy()
    y_ort = sess.run(None, {INPUT_NAME: x})[0]

    with torch.no_grad():
        y_pt = model(dummy).detach().numpy()
        if isinstance(y_pt, (list, tuple)):
            y_pt = y_pt[0]

    cos = np.dot(y_pt.flatten(), y_ort.flatten()) / (
        np.linalg.norm(y_pt) * np.linalg.norm(y_ort) + 1e-12
    )
    print(f"PyTorch vs ORT 余弦: {cos:.6f}")


if __name__ == "__main__":
    export_and_check()
```

## 术语速查

| 术语 | 全称 / 含义 | 一句话直觉 |
|------|------------|-----------|
| ONNX | Open Neural Network Exchange | 中间合同，ORT 验收 + converter 输入 |
| ORT | ONNX Runtime | PC 侧 Golden，不是最终交付物 |
| PTQ | Post-Training Quantization | 训后量化，靠校准集定 scale |
| ATC | Ascend Tensor Compiler | 昇腾 ONNX→`.om`，日志是扯皮依据 |
| 校准集 | Calibration set | 定 INT8 scale，要像线上而不是像训练集 |
| skip_layers | 混合精度跳过层 | 敏感层保持 FP16，别硬量化 |
| Tiling | 分块 | 大张量切块 fit SRAM，影响 MAC 利用率 |
| CPU fallback | 回退 CPU | NPU 不支持的 op 在 CPU 跑，延迟易爆 |
| input.bin | 预处理张量裸二进制 | 双端对比必须同一份字节流 |
| Golden | 基准输出 | PC 标准输入下的参考 output |
| 回滚点 | — | 验收失败退回的上一稳定产物，不是重训 |
